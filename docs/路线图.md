# 卡咔路线图

本文档用于固定卡咔的阶段目标，避免项目一开始就无限堆功能。

## 总体方向

卡咔的长期目标是一个多端接入、多模态、有记忆、有人格、有关系、有状态、有生活痕迹的电子生命系统。

但项目必须分阶段推进：

```text
先能跑
-> 再记得住
-> 再像自己
-> 再看得见听得到
-> 再接入现实硬件
-> 最后逐步学习和进化
```

## 阶段一：第一阶段地基

当前状态：阶段一地基已经通过真实 QQ 测试，并已进入阶段二的正式记忆用于回复阶段。已完成协议层、核心服务、LLM 接入、真实 QQ 文本闭环、SQLite 基础消息/回复记录、群聊普通消息观察记录、最近对话筛选查看、最小输入分析脚本、`memory_candidates` 候选区、候选区查看脚本、程序内置整点自动候选分析、程序内置整点自动候选区复核、正式 `memories` 表、候选区合并脚本、候选区 LLM 复核脚本、正式记忆查看脚本、正式记忆管理脚本、正式记忆检索预览脚本、回复前长期记忆注入、本地 Web 管理台、配置模板、本地自检和第一版长期记忆设计；尚未实现人格状态系统。

目标：

- 项目结构清晰。
- QQ 接入和核心大脑分离。
- 能稳定收发 QQ 消息。
- 有统一消息协议。
- 有基础人格。
- 有基础数据库记录。
- 有最小可运行闭环。

最小闭环：

```text
QQ 消息
-> qq-adapter
-> 统一消息格式
-> kaka-core
-> LLM
-> kaka-core 生成回复
-> qq-adapter
-> QQ 发出回复
-> 数据库记录消息和回复
```

本阶段建议使用：

- Python
- FastAPI
- NoneBot2
- OneBot V11
- SQLite 起步，后续升级 PostgreSQL
- SQLAlchemy
- LLM API

本阶段不做：

- LangGraph
- Redis
- Qdrant
- PyTorch
- AIoT 硬件
- 语音系统
- 图片理解
- 对外部署的复杂管理后台
- 复杂主动回复

完成标准：

- QQ 私聊或群聊能触发卡咔回复。已完成
- `qq-adapter` 不直接写人格和记忆逻辑。已完成
- `kaka-core` 能通过 API 接收统一消息并返回统一响应。已完成
- 数据库里能看到用户、消息和回复记录。已完成
- 能方便查看最近对话记录。已完成

已完成的子项：

- `packages/kaka-protocol` 统一消息协议。
- `services/kaka-core` FastAPI 服务。
- DeepSeek LLM 接入。
- `apps/qq-adapter` OneBot 风格文本事件转换。
- `qq-adapter` 到 `kaka-core` 的 HTTP 客户端。
- `KakaResponse` 到 QQ 文本动作的转换。
- NoneBot2 + OneBot V11 接入。
- NapCat 真实 QQ 文本收发。
- 群聊触发限制。
- SQLite 基础消息和回复记录。
- 群聊普通消息观察记录：写入 `inputs`，不调用模型，不写入 `outputs`。
- `outputs` 输出决策字段：`output_origin / output_reason / no_reply_reason`。
- 最近对话查看脚本，支持按数量、群、用户、日期、场景、是否输出、输出来源和输出原因筛选。
- 最小输入分析脚本 `analyze_inputs.py`，支持规则初筛、批量 LLM 判断、只读预览、候选区写入和 skipped/analyzed 状态更新。
- `memory_candidates` 候选区表和 `show_memory_candidates.py` 查看脚本，支持按候选 ID 查看。
- 程序内置整点自动候选分析，满足门槛时自动写入候选区。
- 程序内置整点自动候选区复核，满足门槛时按批写入正式记忆或更新候选状态。
- 正式 `memories` 表。
- `merge_memory_candidates.py` 候选区合并脚本，支持只读预览、`--apply` 写入和基础去重。
- `review_memory_candidates.py` 候选区 LLM 复核脚本，支持只读预览、`--apply` 写入、拆包重试、关系事实兜底和偏好类型归一化。
- `show_memories.py` 正式记忆查看脚本，支持按记忆 ID 查看。
- `manage_memories.py` 正式记忆管理脚本，支持 `active` / `archived` 切换和硬删除。
- `search_memories.py` 正式记忆检索预览脚本。
- `kaka_core.memory.search` 正式记忆检索模块。
- `kaka_core.context.builder` 回复上下文组装器，当前接入基础人设、长期记忆和当前消息。
- 回复前长期记忆注入，metadata 会记录 `used_memory_ids`。
- 本地 Web 管理台 `apps/web-console`，由 `kaka-core` 托管到 `/admin`。
- 管理 API `/admin/api/*`，覆盖总览、最近对话、输入分析状态调整、候选区状态调整、候选合并、正式记忆管理、检索预览和系统状态。
- `.env.example` 本地配置模板。
- `scripts/doctor.py` 本地自检脚本。
- `docs/长期记忆设计.md` 第一版长期记忆设计。

下一步子项：

- 启动 `kaka-core`，打开 `/admin` 做输入分析、候选区和正式记忆管理的手动验收。
- 启动真实 QQ 链路，观察回复是否自然参考长期记忆。
- 如果回复过度提起旧事，调低 `MEMORY_REPLY_LIMIT` 或提高 `MEMORY_REPLY_MIN_SCORE`。
- 后续再接入短期上下文、关系和情绪状态。

## 阶段二：长期记忆

目标：

- 卡咔能记住人。
- 卡咔能记住重要事件。
- 卡咔不会每次对话都像第一次见面。
- 能根据历史互动调整回复。

重点能力：

- 原始消息记录。
- 短期上下文。
- 长期事件记忆。
- 用户画像。
- 关系记忆。
- 基础记忆抽取。
- 基础记忆检索。

可能引入：

- pgvector
- embedding 模型
- 记忆重要性评分
- 记忆合并和遗忘机制

目标：

- 卡咔有每日生活痕迹。
- 卡咔能写日记。
- 卡咔能根据关系和经历调整语气。

重点能力：

- 心情、精力、困意、好奇心、压力等状态。
- 主人关系和群友关系。
- 语气适配。
- 每日总结。
- 日记生成。

可能引入：

- APScheduler
- 状态数据库表

- 主动行为冷却机制
