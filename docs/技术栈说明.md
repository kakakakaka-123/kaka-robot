# 卡咔技术栈说明

本文档用于解释卡咔会用到的技术分别是什么、解决什么问题、为什么现在需要或暂时不需要。

## Python

Python 是第一阶段的主语言。

它适合这个项目，因为：

- AI、后端、自动化、数据处理生态成熟。
- FastAPI、NoneBot、SQLAlchemy、LangGraph、PyTorch 等核心工具都能用 Python。
- 对学习和调试比较友好。

第一阶段会主要使用 Python。

## TypeScript / React / Vite

TypeScript 适合写 Web 管理台、网页聊天端或桌面端。

当前已经引入到本地 Web 管理台：

```text
apps/web-console
```

当前 Web 界面用途：

- 查看系统总览、正式记忆、提示预演、对话复盘、运行状态和预留扩展入口。
- 管理正式记忆，支持按 ID 倒序排序、每页 50 条分页、新增、编辑、归档、恢复和确认后硬删除。
- 在提示预演页预览当前消息会命中的正式记忆和回复上下文。
- 在对话复盘页分页回查卡咔真实回复过的对话，并查看当次短期上下文和命中的正式记忆。
- 给后续用户画像、关系系统、配置开关、工具日志等管理功能留扩展位置。

最近对话、输入分析、候选区等后台能力仍保留在 API、脚本和数据库中，不再作为日常 Web 页面暴露。

当前仍然不做复杂对外后台，例如登录权限、多用户协作、公网部署和完整运维系统。

## FastAPI

FastAPI 是 Python 的 Web API 框架。

在卡咔中，它负责把 `kaka-core` 做成一个可以被外部调用的核心服务。

例如：

```text
qq-adapter 收到 QQ 消息
-> 请求 kaka-core 的 FastAPI 接口
-> kaka-core 返回回复
```

为什么第一阶段需要：

- 可以让 QQ 接入层和核心大脑分离。
- API 结构清晰。
- 自动生成接口文档。
- 后期 Web、语音、IoT 都能调用同一个核心大脑。
- 当前也托管 `/admin` 页面，并提供 `/admin/api/*` 管理接口。

## NoneBot2

NoneBot2 是 Python 机器人框架。

在卡咔中，它只负责 QQ 接入层，也就是 `apps/qq-adapter`。

它应该做：

- 接收 QQ 私聊和群聊消息。
- 解析 QQ 消息里的文本、图片、表情等。
- 把 QQ 消息转成统一消息协议。
- 调用 `kaka-core`。
- 把 `kaka-core` 的回复发回 QQ。

它不应该做：

- 长期记忆。
- 人格逻辑。
- 关系判断。
- 情绪状态。
- 工具决策。

这些都应该放在 `kaka-core`。

## OneBot V11

OneBot V11 是 QQ 机器人常用的通信协议。

它负责让 NoneBot2 和 QQ 客户端实现通信。

常见组合是：

```text
NoneBot2 + OneBot V11 + NapCat / Lagrange
```

含义是：

- NoneBot2 写机器人逻辑。
- OneBot V11 定义消息通信格式。
- NapCat 或 Lagrange 负责实际连接 QQ。

第一阶段需要它，因为 QQ 是卡咔的第一个入口。

## NapCat / Lagrange

NapCat 和 Lagrange 都可以作为 QQ 接入实现。

它们的作用是：

- 登录 QQ。
- 接收 QQ 消息。
- 按 OneBot 协议把消息交给 NoneBot2。
- 接收 NoneBot2 的发送请求，再发回 QQ。

第一阶段只需要选其中一个能稳定跑通即可，不需要同时支持多个。

## PostgreSQL

PostgreSQL 是主数据库。

它负责保存：

- 用户信息。
- 群信息。
- 原始消息。
- 回复记录。
- 基础记忆。
- 关系数据。
- 状态数据。
- 工具调用记录。

长期为什么不用纯文件或只用 SQLite：

- 卡咔是长期项目，数据会越来越多。
- 后期会有多个服务访问同一份数据。
- PostgreSQL 更适合正式长期运行。
- 后期可以通过 pgvector 扩展向量记忆。

当前阶段已经先用 SQLite 起步：

```text
data/kaka.sqlite3
```

原因：

- 本地开发不需要额外安装数据库服务。
- 可以先验证表结构和记录流程。
- 通过 SQLAlchemy 建模，后续迁移到 PostgreSQL 时业务层改动较小。

所以当前策略是：

```text
SQLite 记录原始对话
-> 字段稳定后
-> PostgreSQL + Alembic
-> 后续 pgvector 长期记忆
```

## SQLAlchemy

SQLAlchemy 是 Python 里常用的数据库工具。

它的作用是让代码用 Python 类和函数操作数据库，而不是到处手写 SQL。

例如未来可以有：

```text
User
Message
Memory
Relationship
```

这些对象当前映射到 SQLite 表，后续可以迁移到 PostgreSQL 表。

第一阶段需要它，因为消息记录、用户记录、基础记忆都要进数据库。

## Alembic

Alembic 是数据库迁移工具，通常和 SQLAlchemy 一起用。

它解决的问题是：

```text
数据库表结构以后会变，怎么安全升级？
```

比如第一版只有 `inputs` 表，后来要加 `memories` 表、`relationships` 表，就可以用 Alembic 管理变化。

第一阶段可以预留或轻量引入，不需要复杂使用。

## LLM API

LLM API 指调用大语言模型的接口。

它负责生成卡咔的自然语言回复。

在第一阶段，流程大概是：

```text
kaka-core 收到统一消息
-> 读取基础人格
-> 读取最近上下文
-> 拼出模型输入
-> 调用 LLM API
-> 得到回复
```

第一阶段需要它，因为没有 LLM，卡咔无法进行自然对话。

## kaka-protocol

`kaka-protocol` 不是外部技术，而是本项目自己的统一消息协议。

它解决的问题是：

```text
QQ、网页、语音、硬件传来的消息格式都不一样，kaka-core 不能为每个平台写一套逻辑。
```

所以要统一成类似这样的结构：

```text
MessageEvent:
  platform: qq / web / voice / iot
  scene_type: private / group / room / device
  scene_id: 群号、房间号或设备 ID
  user_id: 用户唯一 ID
  display_name: 当前显示名
  content_type: text / image / voice / video / sensor
  content: 具体内容
  timestamp: 时间
```

第一阶段应该优先设计它。

## LangGraph

LangGraph 是 Agent 行为编排工具。

它适合管理复杂思考流程，比如：

```text
分析消息
-> 判断是否回复
-> 检索记忆
-> 判断关系
-> 判断是否调用工具
-> 生成回复
-> 检查表达
-> 写入记忆
```

但第一阶段暂时不需要。

原因是第一阶段流程很短：

```text
接收消息 -> 调用 LLM -> 返回回复 -> 记录消息
```

等逻辑变复杂后再引入 LangGraph 更合适。

## Redis

Redis 是缓存和轻量队列工具。

未来可以用于：

- 保存短期上下文。
- 保存临时状态。
- 控制主动回复冷却。
- 做任务队列。
- 支持多个服务之间通信。

第一阶段暂时不需要，因为可以先用数据库和内存完成最小闭环。

## pgvector / Qdrant

pgvector 和 Qdrant 都用于向量检索。

向量检索可以让卡咔根据语义找记忆，而不是只按关键词搜索。

例如用户问：

```text
还记得我之前说过的考试吗？
```

卡咔应该能找到和复习、期末、课程压力相关的记忆。

第一阶段暂时不做复杂向量记忆。阶段二可以先用 `pgvector`，后期数据量大了再考虑 `Qdrant`。

## APScheduler

APScheduler 是 Python 定时任务工具。

未来可以用于：

- 每日总结。
- 写日记。
- 定时整理记忆。
- 定时更新用户画像。
- 定时检查提醒。

第一阶段暂时不需要。阶段三做人格和生活系统时再引入。

## Celery / Dramatiq

Celery 和 Dramatiq 是更重的异步任务队列。

它们适合任务很多、耗时较长、需要后台分布式处理的时候。

第一阶段不需要。等项目变大，APScheduler 不够用时再考虑。

## Whisper / faster-whisper

Whisper 和 faster-whisper 用于语音识别，也就是把语音转成文字。

未来语音入口需要它们。

第一阶段暂时不需要。

## TTS

TTS 是语音合成，也就是把文字变成声音。

未来卡咔如果要用自己的声音说话，需要 TTS。

候选方向包括：

- GPT-SoVITS
- CosyVoice
- Edge TTS
- 云端 TTS

第一阶段暂时不需要。

## ffmpeg

ffmpeg 是音视频处理工具。

未来处理语音、视频、B 站内容时会用到。

第一阶段暂时不需要。

## yt-dlp

yt-dlp 用于下载或解析视频平台内容。

未来 B 站解析和视频总结可能需要它。

第一阶段暂时不需要。

## Playwright

Playwright 是网页自动化工具。

未来用于读取动态网页、网页总结或抓取页面内容。

第一阶段暂时不需要。

## MQTT

MQTT 是物联网常用通信协议。

未来 ESP32、传感器、树莓派、Home Assistant 等设备可以通过 MQTT 和卡咔通信。

第一阶段暂时不需要。阶段五 AIoT 具身化时再引入。

## Docker Compose

Docker Compose 用于管理多个服务一起启动。

未来可以统一启动：

- PostgreSQL
- Redis
- kaka-core
- qq-adapter
- web-console

第一阶段可以先不用复杂部署，但后面接数据库时很适合引入。

## Web 管理台

当前本地 Web 管理台已经实现：

```text
前端：apps/web-console
页面：http://127.0.0.1:8001/admin
API：/admin/api/*
后端路由：services/kaka-core/src/kaka_core/api/admin_routes.py
后端服务：services/kaka-core/src/kaka_core/admin/service.py
```

它解决的问题是：

- 用户后续日常查看和管理数据时不需要自己跑一堆 Python 脚本。
- 正式记忆可以在页面里分页查看、新增、编辑、归档、恢复和删除。
- 提示预演页可以预览命中的正式记忆和真实回复上下文。
- 对话复盘页可以回查卡咔真实回复过的对话、当次短期上下文和命中的正式记忆。
- 后续新增用户画像、关系系统、配置开关、工具日志时有统一入口。

最近对话、输入分析、候选区等后台能力仍保留在 API、脚本和数据库中，不再作为日常 Web 页面暴露。

脚本仍然保留，但定位变成：

```text
开发
测试
批量修复
应急排查
```

## 第一阶段技术选择总结

第一阶段只需要关注：

```text
Python
FastAPI
NoneBot2
OneBot V11
NapCat 或 Lagrange
SQLite
SQLAlchemy
LLM API
kaka-protocol
TypeScript / React / Vite（用于本地 Web 管理台）
```

暂时不要急着引入：

```text
LangGraph
Redis
Qdrant
PyTorch
Whisper
TTS
MQTT
对外部署的复杂管理后台
AIoT 硬件
```

判断标准很简单：

```text
如果它不能帮助第一阶段跑通最小闭环，就先不要加。
```
