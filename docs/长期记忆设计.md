# 长期记忆设计

本文档说明卡咔的第一版长期记忆路线。当前已经实现到“正式记忆落库、检索预览、回复前注入和网页端管理”：`memory_candidates` 可写入和查看，`memories` 表已建立，`merge_memory_candidates.py` 可把候选小批量合并到正式记忆，`review_memory_candidates.py` 可在测试阶段用 LLM 复核候选，`show_memories.py` 可查看正式记忆，`search_memories.py` 可只读预览检索结果，`kaka-core` 回复前会注入少量高分 `active` 记忆，并默认加入同场景最近 30 分钟内最多 8 条短期上下文，本地 Web 管理台 `/admin` 当前用于正式记忆分页、新增、编辑、归档、恢复、硬删除、回复检索和回复上下文预览；输入分析、候选区等后台能力仍保留在 API、脚本和数据库中。

当前阶段的目标不是让卡咔“什么都记住”，而是先建立一套稳定、可解释、可回查的最小记忆规则。

## 设计目标

长期记忆要解决的问题：

- 卡咔能记住用户说过的重要事实。
- 卡咔能记住和用户有关的长期偏好、目标和关系信息。
- 卡咔不会每次对话都像第一次见面。
- 卡咔使用记忆时能追溯来源，而不是凭空编造。

第一版只追求：

```text
原始对话记录
-> 识别值得记住的信息
-> 写入长期记忆表
-> 后续回复时可查询
```

暂时不追求：

- 自动形成复杂人格。
- 自动分析长期情绪变化。
- 向量检索。
- 复杂用户画像。
- 主动冒泡。
- 多轮 Agent 编排。

## 当前已有基础

当前数据库已经有原始记录：

```text
users
scenes
inputs
outputs
memory_candidates
memories
```

这些表的作用：

- `users`：记录外部平台用户，例如 QQ 用户。
- `scenes`：记录对话发生的场景，例如私聊或群聊。
- `inputs`：记录卡咔收到/观察到的输入，包括普通群聊观察消息和触发回复消息。
- `outputs`：记录卡咔对已处理输入形成的输出结果或响应决策；普通观察消息不会写入这张表。
- `memory_candidates`：记录待人工审核的长期记忆候选，来源通过 `source_input_id` 回到原始输入。
- `memories`：记录已经从候选区合并出的正式长期记忆，后续回复检索只应读取 `status=active` 的正式记忆。

第一版长期记忆应该从 `inputs` 和 `outputs` 中抽取，而不是替代原始记录。当前候选区已经从 `inputs.analysis_status=not_analyzed` 的输入中抽取候选。

当前 `inputs` 包含一个后续分析状态字段：

```text
analysis_status   后续分析状态，例如 not_analyzed / analyzed / skipped
```

状态语义：

```text
not_analyzed：还没有被长期记忆分析处理
analyzed：已经处理过，或已经产生候选/被人工标记为已处理
skipped：确认没有长期记忆价值，后续不再分析
```

普通群聊文本的状态通常是：

```text
analysis_status=not_analyzed
```

私聊、@ 和“卡咔”关键词触发的消息通常会同时写入 `inputs` 和 `outputs`。
输出来源和原因记录在 `outputs`：

```text
output_origin   输出来源，例如 passive / active / system
output_reason   输出原因，例如 private / mention / keyword / emotion / scheduled
no_reply_reason 如果 should_reply=false，记录不回复原因，例如 cooldown / safety / merged
```

原则：

```text
原始记录是事实来源
长期记忆是整理后的摘要
```

长期记忆写错时，应该能回到原始消息检查。

## 第一版记忆范围

第一版只记四类内容。

### 用户事实

用户事实是相对稳定、可直接陈述的信息。

可以记：

- 用户的称呼。
- 用户的专业、学校、年级。
- 用户正在做的长期项目。
- 用户明确说过的身份信息。

例子：

```text
用户是物联网工程专业。
用户正在做卡咔项目。
用户主要使用 PyCharm 运行 Python 程序。
```

不应该记：

- 一句玩笑话。
- 语气里的临时情绪。
- 没有明确证据的推测。

### 关系事实

关系事实描述用户和其他人的稳定关系、分工或身份关系。

可以记：

- 用户的室友、导师、同事、队友、家人等关系。
- 用户和其他人的明确身份关系。
- 用户在群里对其他人的分工或关系说明。

例子：

```text
用户的室友小陈负责前端。
用户的导师是王老师。
```

不应该记：

- 用户对卡咔回复风格的偏好。
- 用户希望卡咔怎么说话。
- 没有被用户明确表达或长期稳定表现的信息。

这类“回复风格”偏好应该归到 `stable_preference`，而不是 `relationship_fact`。

### 重要事件

重要事件是对用户后续对话可能有用的阶段性信息。

可以记：

- 考试、课程、项目节点。
- 明确的约定。
- 用户正在处理的重要事情。
- 项目里已经确认完成的重要里程碑。

例子：

```text
2026-05-01，卡咔已跑通 QQ 文本闭环。
2026-05-01，最近对话查看脚本已支持按群、用户、日期和场景筛选。
```

不应该记：

- 普通寒暄。
- 一次性小问题。
- 没有后续价值的闲聊。

### 稳定偏好

稳定偏好是用户反复表达或明确说明的喜好。

可以记：

- 用户偏好用 PyCharm。
- 用户希望文档说明类 Markdown 用中文名。
- 用户希望项目小步推进。
- 用户暂时不急着做人设和复杂主动行为。

不应该记：

- 当天临时想法。
- 一次对话中的随口选择。
- 和长期交互无关的短期状态。

## 不应该记住的内容

第一版明确不记这些内容：

- API Key、密码、Cookie、Token。
- 身份证号、银行卡、住址等高敏感个人信息。
- 用户没有明确要求长期保留的隐私内容。
- 群聊里其他人的敏感信息。
- 模型自己推测出来的性格判断。
- 没有证据的关系结论。
- 明显是玩笑、反话、夸张表达的内容。
- 普通寒暄，例如“早”“晚安”“吃了吗”。

如果不确定是否应该记，第一版策略是：

```text
宁可不记，也不要乱记。
```

## 最小 memories 表设计

当前已实现 `memory_candidates` 候选区和正式 `memories` 表。候选区用于先收集、观察和人工审核，避免模型输出直接污染长期记忆；正式 `memories` 由合并脚本或测试阶段的 LLM 复核脚本写入。

当前 `memory_candidates` 字段：

```text
id
source_input_id
source_user_id
source_scene_id
source_text
candidate_memory
memory_type
confidence
reason
analysis_model
analysis_prompt_version
status
created_at
updated_at
```

当前 `memory_candidates.status` 状态语义：

```text
pending：等待人工审核
approved：已经合并到正式 memories
merged_duplicate：候选和已有正式记忆重复，未新增 memories
rejected：复核后确认不写入正式记忆
```

后续仍可扩展：

```text
merged：人工合并到已有记忆并更新旧记忆
```

当前已新增正式记忆表：

```text
memories
```

当前字段：

```text
id
source_candidate_id
user_id
scene_id
memory_text
normalized_text
memory_type
confidence
source_text
source
status
merge_reason
created_at
updated_at
```

字段说明：

- `source_candidate_id`：来源候选，可追溯到 `memory_candidates`，再追溯到原始 `inputs`。
- `user_id`：记忆归属用户。
- `scene_id`：来源场景，可为空，后续检索时可区分私聊/群聊。
- `memory_text`：正式记忆正文。
- `normalized_text`：用于基础去重的归一化正文。
- `memory_type`：记忆类型，例如 `user_fact / relationship_fact / important_event / stable_preference`。
- `confidence`：候选分析给出的置信度。
- `source_text`：来源消息文本快照。
- `source`：来源，当前为 `candidate`。
- `status`：正式记忆状态，当前使用 `active`。
- `merge_reason`：合并原因。
- `user_id`：这条记忆属于哪个用户。
- `scene_id`：这条记忆是否和某个群或私聊场景有关。
- `source_input_id`：这条记忆从哪条原始输入抽取出来。
- `memory_type`：例如 `user_fact`、`relationship_fact`、`important_event`、`stable_preference`。
- `content`：可直接读懂的中文记忆内容。
- `confidence`：模型或规则对这条记忆的确信程度，第一版可以是 0 到 1。
- `importance`：重要性，第一版可以是 1 到 5。
- `status`：例如 `active`、`archived`、`rejected`。
- `metadata`：保存抽取原因、关键词、模型信息等。

第一版不建议直接上复杂多表结构。

原因：

```text
先验证“能不能稳定抽取有用记忆”
再考虑拆分用户画像、事件、关系、偏好等细表
```

## 记忆类型

第一版只使用四种类型：

```text
user_fact
relationship_fact
important_event
stable_preference
```

### user_fact

用户事实。

例子：

```text
用户是物联网工程专业。
用户主要使用 PyCharm。
```

### relationship_fact

用户和其他人的明确关系事实。

例子：

```text
用户的室友小陈负责前端。
用户的导师是王老师。
```

### important_event

重要事件或项目节点。

例子：

```text
2026-05-01，卡咔完成 QQ 文本闭环和基础记录工具。
```

### stable_preference

稳定偏好。

例子：

```text
用户希望说明类 Markdown 文档使用中文文件名。
用户希望 Python 注释和 docstring 优先使用中文。
```

## 抽取时机

当前项目已经采用“半自动候选抽取”：

```text
规则初筛 skipped / candidate / not_sure
-> not_sure 批量发给记忆模型判断
-> candidate 写入 memory_candidates
-> skipped 标记 inputs.analysis_status=skipped
-> candidate 来源 input 标记 inputs.analysis_status=analyzed
-> LLM error 或缺结果保持 not_analyzed
```

手动方式：

```powershell
python services/kaka-core/scripts/analyze_inputs.py --llm-batch
python services/kaka-core/scripts/analyze_inputs.py --llm-batch --write-candidates
python services/kaka-core/scripts/show_memory_candidates.py
python services/kaka-core/scripts/show_memory_candidates.py --ids 12,13
python services/kaka-core/scripts/merge_memory_candidates.py --limit 20
python services/kaka-core/scripts/merge_memory_candidates.py --limit 20 --apply
python services/kaka-core/scripts/merge_memory_candidates.py --ids 12,13 --apply
python services/kaka-core/scripts/review_memory_candidates.py --limit 20
python services/kaka-core/scripts/review_memory_candidates.py --limit 20 --apply
python services/kaka-core/scripts/review_memory_candidates.py --ids 12,13 --apply
python services/kaka-core/scripts/show_memories.py --limit 20
python services/kaka-core/scripts/show_memories.py --ids 12,13
python services/kaka-core/scripts/search_memories.py --user QQ号 --text "当前消息文本"
```

当前日常方式：

```text
打开 http://127.0.0.1:8001/admin
-> 在正式记忆页分页查看、新增、编辑、归档、恢复或确认后硬删除
-> 在回复检索页预览回复前可能命中的 active 记忆和真实回复上下文
-> 输入分析和候选区处理改用管理 API、脚本或数据库
```

如果用数据库可视化软件先看表，开发或应急场景也可以在这些脚本顶部填写 `PYCHARM_INPUT_IDS`、`PYCHARM_CANDIDATE_IDS`、`PYCHARM_MEMORY_IDS`、`PYCHARM_APPLY` 等简单配置，再右键运行；默认都先预览，不写库。

自动方式：

```text
kaka-core 启动后，如果 MEMORY_AUTO_ANALYSIS_ENABLED=true：
1. 等到下一个整点检查。
2. 之后每个整点检查一次。
3. not_analyzed >= 50 时执行。
4. 每次最多 2 轮，每轮最多 50 条。
```

第一版仍然不建议让模型输出直接进入正式长期记忆。当前做法是先进入 `memory_candidates.pending`，再由 `merge_memory_candidates.py` 小批量预览和合并；如果测试阶段没有人工审核时间，也可以临时用 `review_memory_candidates.py` 让 LLM 代替人工做复核。

历史设计建议是先做手动或半自动抽取，不要一开始就每条消息都自动抽取。

推荐顺序：

1. 从最近对话查看脚本中人工观察哪些内容值得记。
2. 写一个最小抽取函数或脚本，只处理指定消息。
3. 抽取结果先打印出来，不立刻写入数据库。
4. 确认格式稳定后，先写入 `memory_candidates`。
5. 用 `show_memory_candidates.py` 查看 pending 候选。
6. 用 `merge_memory_candidates.py --limit 20` 预览合并，或者用 `review_memory_candidates.py --limit 20` 做 LLM 复核。
7. 确认后用 `--apply` 写入正式 `memories`。
8. 用 `show_memories.py` 查看正式记忆。
9. 用 `search_memories.py` 预览当前消息会命中哪些正式记忆。
10. 最后再考虑在回复时查询正式记忆。

第一版不建议：

- 每条触发消息都自动写长期记忆。
- 群聊普通消息未经筛选就自动进入记忆。
- 未经检查就把模型输出直接写入记忆。

## 抽取规则

抽取时应该问四个问题：

```text
1. 这句话是否包含长期有用的信息？
2. 这条信息是否来自用户明确表达？
3. 这条信息以后回复时是否可能用得上？
4. 这条信息是否安全、合适、不过度隐私？
```

只有四个问题都倾向于“是”，才进入候选记忆。

候选记忆还需要判断：

- 是否和已有记忆重复。
- 是否只是已有记忆的改写。
- 是否和已有记忆冲突。
- 是否需要更新旧记忆，而不是新增一条。

## 去重和更新

第一版去重可以先用简单规则：

- 同一用户。
- 同一记忆类型。
- 内容高度相似。
- 来源时间接近或表达意思相同。

遇到重复时，不新增，优先更新：

```text
updated_at
confidence
importance
metadata
```

当前已实现的基础去重更保守：

```text
同一用户 + 同一记忆类型 + normalized_text 完全相同
```

重复候选不会新增正式记忆，会把候选状态标记为 `merged_duplicate`。

遇到冲突时，不直接覆盖旧记忆。

例如：

```text
旧记忆：用户主要使用 PyCharm。
新信息：用户开始用命令行运行服务。
```

这不一定是冲突，可能只是新增习惯。第一版应保守处理，可以先记录为候选，人工确认后再更新。

## 使用记忆的方式

第一版生成回复时，不需要复杂检索。

可以先按用户和场景查询少量记忆：

```text
当前用户 active 记忆
当前场景相关 active 记忆
importance 较高的记忆
最近使用过但不过期的记忆
```

然后拼进大模型输入。

注意：

- 记忆只能作为上下文，不能强迫模型每次都提。
- 不要在无关场景主动暴露敏感或私人记忆。
- 不要为了展示“我记得”而生硬提起旧事。

## 和当前原始记录的关系

当前 `inputs` 和 `outputs` 仍然是最重要的数据源。

长期记忆不是聊天记录的替代品。

关系应该是：

```text
inputs / outputs
-> 事实来源
-> 抽取候选记忆
-> memory_candidates
-> 人工审核 / 合并
-> memories
-> 回复时作为上下文
```

如果记忆有问题，应优先检查：

```text
source_input_id
```

这样可以知道它从哪条原始输入来。

后续实现时建议按这个顺序：

1. 已完成 `memory_candidates` 数据模型。
2. 已完成候选区写入和查看脚本。
3. 已完成程序内置整点自动候选分析。
4. 已完成正式 `MemoryRecord` / `memories` 表。
5. 已完成候选区合并脚本 `merge_memory_candidates.py`。
6. 已完成候选区 LLM 复核脚本 `review_memory_candidates.py`。
7. 已完成正式记忆查看脚本 `show_memories.py`。
8. 已完成正式记忆只读检索预览脚本 `search_memories.py`。
9. 已用合成数据跑通过一轮长期记忆 E2E 回放。
10. 已完成回复前少量高置信度正式记忆注入。
11. 已完成程序内置整点自动候选区复核。
12. 已完成正式记忆管理脚本 `manage_memories.py`，支持归档、恢复和硬删除。
13. 已完成本地 Web 管理台 `/admin`，当前日常页面支持总览、正式记忆管理、回复检索、运行状态和预留扩展入口；输入分析和候选区能力保留在 API、脚本和数据库中。
14. 已完成正式记忆页分页、按 ID 稳定排序、手动新增和单条编辑。
15. 已完成回复上下文预览，便于检查真实回复前注入到模型的 System Prompt、User Prompt 和 metadata。
16. 已完成第一版短期上下文注入，默认读取同场景最近 30 分钟内最多 8 条输入记录，总字符上限 1200，并排除当前消息。

暂时不要：

- 接入向量库。
- 接入复杂嵌入模型。
- 做自动遗忘算法。
- 做复杂关系图谱。
- 做主动行为系统。

## 第一版完成标准

第一版长期记忆功能可以认为完成，当满足：

```text
给定一条真实对话消息
-> 能判断是否值得记
-> 能生成一条候选记忆
-> 能追溯来源消息
-> 能写入 memories 表
-> 能在后续对话中查询并作为上下文使用
```

当前已满足到“能查询正式 memories，并在回复前作为上下文进入 prompt”。候选分析、候选区复核和正式记忆管理已经可以自动或半自动运行；Web 管理台已经成为正式记忆日常管理入口；短期上下文已经能让回复参考最近对话。后续主要通过真实 QQ 对话观察使用效果。

同时必须满足：

- 不保存密钥和高敏感隐私。
- 不把普通寒暄写进长期记忆。
- 不把模型推测当成事实。
- 能人工查看和删除错误记忆。

## 当前结论

卡咔的长期记忆第一版应该是：

```text
小而稳
可回查
可人工检查
```
