# 下一个对话框快速上下文

本文档给新的对话框快速恢复项目进度。建议新对话框优先阅读本文件，再阅读 `KAKA_HANDOFF.md` 和 `docs/开发运行说明.md`。

## 项目一句话

卡咔不是普通 QQ 机器人，而是一个以 `kaka-core` 为核心大脑、未来可接入 QQ/Web/语音/AIoT 的 AI 人格体项目。

核心原则：

```text
卡咔不是 QQ 机器人，QQ 只是卡咔的一个身体。
```

## 用户偏好

- 使用中文交流。
- 用户希望理解每个技术是干什么的，但不希望一开始就被大量复杂代码压住。
- 用户希望开发者负责架构和代码，同时解释“为什么这样做”。
- 用户希望项目保持清晰、可学习、可维护，不要盲目堆功能。
- Python 文件里的注释尽量用中文，关键处适当解释。
- 用户用 PyCharm，喜欢可以直接运行的 `run.py` / `bot.py` 入口。
- 后续新增 Python 注释和 docstring 优先使用中文；技术名词、接口字段、环境变量名和第三方库名称可以保留原文。
- 用户不习惯记命令行参数；开发辅助脚本需要在文件顶部写清楚 PyCharm 右键运行方式、可用参数和常用例子。

### kaka-core

位置：

```text
services/kaka-core
```

作用：卡咔核心大脑 API。

已完成：

- FastAPI 应用。
- `GET /health`
- `POST /v1/chat`
- `POST /v1/observe`
- `run.py` 本地启动入口。
- DeepSeek LLM 接入。
- OpenAI-compatible `LLMClient`。
- 预留 `LLMRouter`，以后可按任务选择不同模型。
- SQLite 基础消息和回复记录。
- 最近对话查看脚本：支持 `--limit`、`--group`、`--user`、`--date`、`--private`、`--group-chat`、`--replied-only`、`--observed-only`、`--origin`、`--reason`。
- 输入分析脚本：支持规则初筛、批量 LLM 判断、候选区写入和 skipped/analyzed 状态更新。
- 候选区查看脚本：支持按状态、类型、群、用户、日期筛选。
- 候选区合并脚本：默认只读预览，加 `--apply` 后把 pending 候选合并到正式 `memories`；PyCharm 可直接填 `PYCHARM_CANDIDATE_IDS`。
- 候选区 LLM 复核脚本：默认只读预览，加 `--apply` 后按 approve/reject/duplicate 写入正式记忆或更新候选状态。
- 正式记忆查看脚本：`show_memories.py` 可查看已经合并出的正式 `memories`，也可直接按 `memories.id` 查。
- 正式记忆管理脚本：`manage_memories.py` 可把记忆切到 `active` / `archived`，或在确认错误、垃圾、敏感时硬删除。
- 正式记忆检索预览脚本：`search_memories.py` 可按当前用户和当前消息，只读预览会命中的少量正式记忆。
- 自动候选分析：可通过 `.env` 开启，kaka-core 启动后按整点检查并在满足门槛时自动处理。
- 自动候选区复核：可通过 `.env` 开启，kaka-core 启动后按整点检查；`pending >= 20` 时每次最多复核 10 条。
- 基础人设 Prompt：默认从 `prompts/kaka_persona.md` 读取；可通过 `KAKA_PERSONA_PROMPT_PATH` 改路径；文件缺失或为空时会回退到内置基础人设。
- 短期上下文：默认开启，回复前读取同场景最近 30 分钟内最多 20 条输入记录，总字符上限 1200，并排除当前消息。
- 关系上下文：默认开启，回复前用 `KAKA_OWNER_USER_IDS`、历史输入数、最近 7 天输入数和 active 正式记忆数粗分 `owner / familiar / regular / stranger`。阈值偏保守：familiar 为历史输入 >=100 或最近 7 天 >=30 或 active 记忆 >=8；regular 为历史输入 >=30 或最近 7 天 >=10 或 active 记忆 >=3。
- 本地 Web 管理台：`http://127.0.0.1:8001/admin`，前端在 `apps/web-console`，API 在 `/admin/api/*`；日常正式记忆管理优先走网页。
- 管理 API 当前覆盖：总览、最近对话、输入分析预览、输入状态调整、候选区列表、候选状态调整、候选合并、正式记忆列表、正式记忆分页、新增、编辑、归档、恢复、删除、记忆检索预览、回复上下文预览和系统状态。

当前默认数据库文件：

```text
data/kaka.sqlite3
```

当前表：

```text
users
scenes
inputs
outputs
memory_candidates
memories
```

表语义：

```text
inputs：卡咔收到/观察到的输入，包含 analysis_status。
outputs：卡咔对已处理输入形成的输出结果或响应决策，包含 output_origin、output_reason、no_reply_reason；普通观察消息不会写入 outputs。
memory_candidates：待人工审核的长期记忆候选，来源可追溯到 source_input_id。
memories：已经从候选区合并出的正式长期记忆，后续检索只应读取 active 记忆。
```

状态语义：

```text
inputs.analysis_status:
  not_analyzed：还没有被长期记忆分析处理
  analyzed：已经处理过，或已经产生候选/被人工标记为已处理
  skipped：确认没有长期记忆价值，后续不再分析

memory_candidates.status:
  pending：等待审核或合并
  approved：已经合并进正式 memories
  rejected：确认不进入正式记忆
  merged_duplicate：和已有正式记忆重复，没有新增 memories

memories.status:
  active：参与回复前检索
  archived：保留但不参与回复
```

`.gitignore` 已忽略本地 SQLite 文件，避免误提交真实聊天记录。

当前阶段状态：

```text
第一阶段地基已经通过真实 QQ 测试；当前已进入正式长期记忆用于回复的阶段。memories 表、自动候选分析、自动候选区复核、正式记忆查看/管理/检索脚本、回复前长期记忆注入、回复前短期上下文注入、回复前关系上下文注入和本地 Web 管理台都已完成。Web 日常页面当前只保留总览、正式记忆、回复检索、运行状态和预留扩展入口；正式记忆页已支持分页、新增、编辑、归档、恢复和硬删除，输入分析和候选区仍走 API、脚本和数据库。
```

建议顺序：

1. 启动 `kaka-core`，打开 `http://127.0.0.1:8001/admin` 做一次手动验收。
2. 检查左侧导航的总览、正式记忆、回复检索、运行状态和预留扩展是否正常。
3. 在正式记忆页复查分页、新增、编辑、归档、恢复和确认后硬删除。
4. 再启动 `qq-adapter`，用真实 QQ 对话观察卡咔是否自然参考长期记忆。
5. 回查响应 metadata 中的 `used_memory_ids`、`memory_count` 和 `memory_injection_enabled`。
6. 回查响应 metadata 中的 `short_context_count` 和 `short_context_input_ids`，确认短期上下文没有串场景。
7. 回查响应 metadata 中的 `relationship_level`、`relationship_is_owner` 和关系计数，确认主人、熟人、普通群友和新人区分合理。
8. 如果回复过度提起旧事，优先调 `MEMORY_REPLY_LIMIT` 和 `MEMORY_REPLY_MIN_SCORE`；如果被最近闲聊干扰，优先调小 `SHORT_CONTEXT_LIMIT` 或关闭 `SHORT_CONTEXT_ENABLED`。
8. 平时用 `/admin` 检查 `memories`，不合适的记忆优先归档，确认错误、垃圾或敏感再硬删除；确需手动补记或修正时用新增/编辑。

## 暂时不要做

在原始对话记录稳定可查前，暂时不要做：

- 图片理解。
- 表情包系统。
- 语音识别和语音合成。
- 大量回复 prompt 注入。
- 用户画像。
- 复杂关系系统和好感度分数。
- 群聊复杂主动回复。
- LangGraph。
- AIoT。

这些都可以后续做，但现在会干扰第一阶段地基收尾。
