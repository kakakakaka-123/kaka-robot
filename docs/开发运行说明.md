### 桌宠核心连接配置

桌宠默认读取 `KAKA_CORE_BASE_URL` 连接本机 `kaka-core`。如果需要单独覆盖桌宠连接，可以配置：

```env
KAKA_DESKTOP_CORE_ADDR=127.0.0.1:8001
KAKA_DESKTOP_CORE_HOST_HEADER=127.0.0.1:8001
KAKA_DESKTOP_USER_ID=你的QQ号
KAKA_DESKTOP_DISPLAY_NAME=主人
KAKA_DESKTOP_SCENE_ID=desktop-local
```

规则：

- `KAKA_DESKTOP_CORE_ADDR` 为空时，会从 `KAKA_CORE_BASE_URL` 提取地址。
- `KAKA_DESKTOP_USER_ID` 为空时，会优先使用 `KAKA_OWNER_USER_IDS` 里的第一个 ID。
- `KAKA_DESKTOP_DISPLAY_NAME` 和 `KAKA_DESKTOP_SCENE_ID` 只影响桌宠发给 `kaka-core` 的 `MessageEvent` 身份和场景。
- 这些配置只改变桌宠本地入口，不改变 QQ 适配器和 Web 管理台。

## 数据库记录

当前 `kaka-core` 会把基础消息和回复记录写入 SQLite。

默认数据库文件：

```text
D:\Python\AgentRobot\kaka-v2\data\kaka.sqlite3
```

当前表：

```text
users
scenes
inputs
outputs
memory_candidates
memories
auto_job_runs
```

当前表语义：

```text
users：外部平台用户。
scenes：私聊、群聊等消息场景。
inputs：卡咔收到/观察到的输入，包含 analysis_status。
outputs：卡咔对已处理输入形成的输出结果或响应决策，包含 output_origin、output_reason、no_reply_reason；普通观察消息不会写入 outputs。
memory_candidates：长期记忆候选区，候选先在这里等待审核、拒绝、标记重复或合并。
memories：正式长期记忆表，回复前只检索 active 记忆。
auto_job_runs：自动候选分析和自动候选复核的运行记录。
```

当前状态语义：

```text
inputs.analysis_status：
  not_analyzed：还没有被长期记忆分析处理
  analyzed：已经处理过，或已经产生候选/被人工标记为已处理
  skipped：确认没有长期记忆价值，后续不再分析

memory_candidates.status：
  pending：等待审核或合并
  approved：已经合并进正式 memories
  rejected：确认不进入正式记忆
  merged_duplicate：和已有正式记忆重复，没有新增 memories

memories.status：
  active：参与回复前检索
  archived：保留但不参与回复
```

时间说明：

```text
created_at 当前按 UTC 存储。
在 Navicat 里直接看会比北京时间少 8 小时。
例如数据库里 2026-04-30 17:40:49，对应北京时间 2026-05-01 01:40:49。
```

数据库文件包含真实聊天记录，已经被 `.gitignore` 忽略，不应该提交。

如果需要换数据库地址，可以设置：

```env
DATABASE_URL=sqlite:///D:/Python/AgentRobot/kaka-v2/data/kaka.sqlite3
```

## 分析未处理输入

可以使用最小分析脚本查看还没有进入后续分析流程的输入：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py
```

脚本默认只读预览，不修改数据库。

如果在 PyCharm 中右键运行，不需要配置 Parameters。直接修改脚本顶部的：

```python
PYCHARM_INPUT_IDS = ""          # 已知 inputs.id 时填写，例如 "12,13"
PYCHARM_LIMIT = 30
PYCHARM_GROUP_ID = ""
PYCHARM_USER_ID = ""
PYCHARM_LLM_BATCH = False
PYCHARM_WRITE_CANDIDATES = False
PYCHARM_MARK_SKIPPED = False
```

默认只读预览。`PYCHARM_WRITE_CANDIDATES=True` 必须配合 `PYCHARM_LLM_BATCH=True`；确认输出没问题后再打开写库开关。

常用筛选：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --limit 50
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --group 1073224364
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --user 1419825488
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --date 2026-05-01
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --private
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --group-chat
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --llm-batch
```

当前分类：

```text
skipped：明显无价值或占位消息，例如寒暄、测试消息、@ 占位
candidate：可能值得后续进入记忆候选，例如身份、偏好、项目长期信息
not_sure：普通聊天，暂时不自动跳过
```

批量调用记忆模型做只读判断：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --limit 50 --llm-batch
```

确认输出没问题后，写入候选区：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --limit 50 --llm-batch --write-candidates
```

`--write-candidates` 会修改数据库：

```text
LLM/rule candidate -> 写入 memory_candidates，来源 input 标记 analyzed
LLM/rule skipped   -> 来源 input 标记 skipped
LLM error / 缺结果 -> 保持 not_analyzed
```

如果确认明显无价值记录可以跳过，可以加：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --mark-skipped
```

`--mark-skipped` 只会把脚本判定为 `skipped` 的记录更新为：

```text
analysis_status=skipped
```

不会写入长期记忆，也不会修改 `outputs`。

查看候选区：


`search_memories.py` 只读读取 `memories`，按当前用户、当前消息、可选场景打分，打印少量命中记忆和命中原因，不会修改数据库，也不会影响当前聊天回复。

PyCharm 里直接填：

```python
PYCHARM_USER_ID = "QQ号"
PYCHARM_TEXT = "当前消息文本"
PYCHARM_GROUP_ID = ""   # 群聊填群号；私聊则把 PYCHARM_PRIVATE 改 True
PYCHARM_PRIVATE = False
```

如果在 `.env` 开启自动候选分析：

```env
MEMORY_AUTO_ANALYSIS_ENABLED=true
MEMORY_AUTO_ANALYSIS_TRIGGER_COUNT=50
MEMORY_AUTO_ANALYSIS_BATCH_LIMIT=50
MEMORY_AUTO_ANALYSIS_MAX_RUNS_PER_CHECK=2
```

`kaka-core` 启动后不会立刻分析，会读取本机时间，等到下一个整点检查；之后每个整点检查一次。只有 `not_analyzed >= 50` 时才调用大模型，每个整点最多跑两轮。运行状态页可以手动触发一次，手动触发会绕过数量门槛。

如果在 `.env` 开启自动候选区复核：

```env
MEMORY_AUTO_REVIEW_ENABLED=true
MEMORY_AUTO_REVIEW_TRIGGER_COUNT=20
MEMORY_AUTO_REVIEW_BATCH_SIZE=10
MEMORY_AUTO_REVIEW_MAX_RUNS_PER_CHECK=1
```

`kaka-core` 启动后同样等到下一个整点检查。只有 `memory_candidates.pending >= 20` 时才调用大模型；每批 10 条，每个整点最多跑一批。运行状态页可以手动触发一次，手动触发会绕过数量门槛。

自动候选分析和自动候选区复核都使用跨进程任务锁。如果本机还有旧的 `kaka-core/run.py` 进程在运行，它到整点仍会继续检查自动任务；没有任何 `kaka-core` 进程运行时，自动任务不会自己执行。
