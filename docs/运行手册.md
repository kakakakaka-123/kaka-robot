# 卡咔运行与排错清单

这份文档只解决一件事：当前项目已经能跑起来以后，如何稳定启动、测试和排查问题。

当前链路是：

```text
QQ
-> NapCat
-> OneBot V11 反向 WebSocket
-> NoneBot2 / qq-adapter
-> kaka-core
-> DeepSeek
-> QQ 回复
-> SQLite 记录观察消息和回复
```

核心原则：

```text
卡咔不是 QQ 机器人，QQ 只是卡咔的一个身体。
```

所以排错时要分清楚：

- NapCat 负责 QQ 和 OneBot 连接。
- `qq-adapter` 负责接收 QQ 事件、判断回复或观察、调用核心服务。
- `kaka-core` 负责生成回复、观察记录和写入数据库。

## 当前端口

```text
kaka-core: http://127.0.0.1:8001
qq-adapter / NoneBot2: http://127.0.0.1:8081
NapCat reverse WebSocket: ws://127.0.0.1:8081/onebot/v11/ws
```

NapCat 的 OneBot V11 反向 WebSocket 地址应填写：

```text
ws://127.0.0.1:8081/onebot/v11/ws
```

## 正常启动顺序

建议按这个顺序启动：

1. 启动 NapCat，并确认当前 QQ 已登录。
2. 启动 `kaka-core`。
3. 启动 `qq-adapter`。
4. 在 QQ 私聊或测试群里发消息验证。

`kaka-core` 启动入口：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\run.py
```

PyCharm 可直接运行：

```text
services/kaka-core/run.py
```

启动成功后可以打开：

```text
http://127.0.0.1:8001/docs
```

本地网页管理台：

```text
http://127.0.0.1:8001/admin
```

管理台用于日常查看总览、正式记忆、回复检索预览和系统状态。正式记忆支持分页、新增、编辑、归档、恢复和确认后硬删除；最近对话、输入分析、候选区等后台能力保留在管理 API、脚本和数据库中。写库动作仍保留按钮确认；不要把 `kaka-core` 端口暴露到公网。

当前日常数据管理优先使用这个网页。下面的脚本主要留给开发、测试、批量修复和应急排查。

`qq-adapter` 启动入口：

```powershell
.\.venv\Scripts\python.exe .\apps\qq-adapter\bot.py
```

PyCharm 可直接运行：

```text
apps/qq-adapter/bot.py
```

## 当前触发规则

```text
私聊：所有文本消息都会触发卡咔回复。
群聊：只有 @卡咔 或消息里包含“卡咔”才触发。
群聊普通消息：不触发回复，但会写入 inputs 作为观察记录。
```

这是当前阶段的刻意设计：群聊里不强制每句话都叫卡咔，但卡咔仍然能保留完整文本上下文，后续才能做用户画像、情绪识别和记忆候选判断。

## 基础测试用例

每次改完关键逻辑后，建议按这个顺序测试：

```text
1. 私聊发送普通文本：应该回复。
2. 群聊发送普通文本：不应该回复，但最近对话脚本应能查到观察记录。
3. 群聊发送包含“卡咔”的文本：应该回复。
4. 群聊 @卡咔：应该回复。
5. 关闭 kaka-core 后触发：qq-adapter 应提示核心服务暂时没连上，不应刷 traceback。
6. 重启 kaka-core 后再次触发：应该恢复回复。
7. 查看数据库最近对话：普通观察和触发回复都应能查到。
```


脚本会把数据库中的 UTC 时间转换为北京时间显示。

默认数据库位置：

```text
D:\Python\AgentRobot\kaka-v2\data\kaka.sqlite3
```

当前主要表：

```text
users
scenes
inputs
outputs
memory_candidates
memories
```

表语义：

```text
users：外部平台用户。
scenes：私聊、群聊等消息场景。
inputs：卡咔收到/观察到的输入，包含 analysis_status。
outputs：卡咔对已处理输入形成的输出结果或响应决策，包含 output_origin、output_reason、no_reply_reason；普通观察消息不会写入 outputs。
memory_candidates：长期记忆候选区。
memories：已经从候选区合并出的正式长期记忆。
```

当前状态语义：

```text
inputs.analysis_status：
  not_analyzed：还没有被长期记忆分析处理
  analyzed：已经处理过，或已经产生候选/被人工标记为已处理
  skipped：确认没有长期记忆价值，后续不再分析

memory_candidates.status：
  pending：等待审核或合并
  approved：已经合并进正式 memories
  rejected：确认不进入正式记忆
  merged_duplicate：和已有正式记忆重复，没有新增 memories

memories.status：
  active：参与回复前检索
  archived：保留但不参与回复
```

网页管理台当前支持：

```text
总览：查看 inputs、outputs、候选区和正式记忆数量。
正式记忆：按 ID 稳定排序，每页 50 条，支持筛选、新增、编辑、归档、恢复或确认后硬删除。
回复检索：预览当前消息会命中的 active 记忆，并查看真实回复上下文预览。
运行状态：查看关键配置和数据库连接概况。
预留扩展：保留后续用户画像、关系、工具日志和配置入口。
```

输入分析、候选区和候选合并仍保留在管理 API、脚本和数据库中，不再作为日常 Web 页面暴露。

## 分析未处理输入

使用最小输入分析脚本：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py
```

PyCharm 右键运行时，不需要配置 Parameters。直接改脚本顶部的简单配置即可：

```python
PYCHARM_INPUT_IDS = ""          # 已知 inputs.id 时填写，例如 "12,13"
PYCHARM_LIMIT = 30
PYCHARM_GROUP_ID = ""
PYCHARM_USER_ID = ""
PYCHARM_LLM_BATCH = False
PYCHARM_WRITE_CANDIDATES = False
PYCHARM_MARK_SKIPPED = False
```

默认只读预览。`PYCHARM_WRITE_CANDIDATES=True` 必须配合 `PYCHARM_LLM_BATCH=True`；确认输出没问题后再打开写库开关。

它会扫描：

```text
inputs.analysis_status=not_analyzed
```

默认只读打印，不修改数据库。

常用筛选：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --limit 50
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --group 1073224364
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --user 1419825488
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --date 2026-05-01
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --group-chat
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --llm-batch
```

只读批量调用记忆模型：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --limit 50 --llm-batch
```

确认结果后写入候选区：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --limit 50 --llm-batch --write-candidates
```

写入规则：

```text
candidate -> memory_candidates，来源 input 标记 analyzed
skipped   -> 来源 input 标记 skipped
error     -> 保持 not_analyzed
```

如果确认明显无价值记录可以跳过，再使用：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\analyze_inputs.py --mark-skipped
```

`--mark-skipped` 只会把脚本判定为 `skipped` 的记录更新为 `analysis_status=skipped`，不会写长期记忆。

查看候选区：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memory_candidates.py
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memory_candidates.py --status all --limit 50
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memory_candidates.py --ids 12,13
```

PyCharm 里可以直接填：

```python
PYCHARM_CANDIDATE_IDS = "候选ID"
PYCHARM_STATUS = ""  # 填了 ID 时默认查 all；不填 ID 时默认查 pending
```

合并候选到正式记忆：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\merge_memory_candidates.py --limit 20
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\merge_memory_candidates.py --limit 20 --apply
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\merge_memory_candidates.py --ids 12,13 --apply
```

第一条只读预览，不修改数据库。第二条会写入 `memories`，并把成功合并的候选标记为 `approved`，重复候选标记为 `merged_duplicate`。

如果已经在数据库可视化软件里确认了候选 ID，更推荐打开脚本填：

```python
PYCHARM_CANDIDATE_IDS = "候选ID"
PYCHARM_APPLY = False  # 先预览；确认后改 True
```

如果测试阶段没有时间人工检查候选区，可以用 LLM 临时代替人工复核：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\review_memory_candidates.py --limit 20
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\review_memory_candidates.py --limit 20 --apply
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\review_memory_candidates.py --ids 12,13 --apply
```

不加 `--apply` 时只读预览；加 `--apply` 后会按模型复核结果写入正式 `memories`，并把候选标记为 `approved`、`rejected` 或 `merged_duplicate`。

PyCharm 里同样可以只填：

```python
PYCHARM_CANDIDATE_IDS = "候选ID"
PYCHARM_APPLY = False  # 先预览；确认后改 True
```

查看正式记忆：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memories.py --limit 20
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memories.py --status all --limit 50
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memories.py --type user_fact --limit 30
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memories.py --ids 12,13
```

`show_memories.py` 只读读取 `memories`，不会修改数据库。

PyCharm 里可以直接填：

```python
PYCHARM_MEMORY_IDS = "记忆ID"
PYCHARM_STATUS = ""  # 填了 ID 时默认查 all；不填 ID 时默认查 active
```

管理正式记忆状态或硬删除：

如果已经在数据库可视化软件里看到 `memories.id`，可以直接打开：

```text
services/kaka-core/scripts/manage_memories.py
```

在文件顶部填写：

```python
PYCHARM_MEMORY_IDS = "记忆ID"
PYCHARM_ACTION = "archive"   # archive / restore / delete
PYCHARM_APPLY = False        # 先预览；确认后改 True
```

命令行方式仍然可用：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\manage_memories.py --id 记忆ID --status archived
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\manage_memories.py --id 记忆ID --status archived --apply
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\manage_memories.py --id 记忆ID --status active --apply
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\manage_memories.py --id 记忆ID --delete --apply --yes
```

`active` 会参与回复检索，`archived` 会保留记录但不参与回复。硬删除只用于确认错误、垃圾或敏感记忆。

预览正式记忆检索：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\search_memories.py --user QQ号 --text "当前消息文本"
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\search_memories.py --user QQ号 --group 群号 --text "当前消息文本"
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\search_memories.py --user QQ号 --private --text "当前消息文本"
```

`search_memories.py` 只读读取 `memories`，不会修改数据库。聊天回复前也会复用同一套检索逻辑，因此这个脚本可用于预览回复时可能注入哪些正式记忆。

PyCharm 里直接填：

```python
PYCHARM_USER_ID = "QQ号"
PYCHARM_TEXT = "当前消息文本"
PYCHARM_GROUP_ID = ""   # 群聊填群号；私聊则把 PYCHARM_PRIVATE 改 True
PYCHARM_PRIVATE = False
```

回复时长期记忆注入可以在根目录 `.env` 中配置：

```env
MEMORY_REPLY_INJECTION_ENABLED=true
MEMORY_REPLY_LIMIT=5
MEMORY_REPLY_MIN_SCORE=1.0
MEMORY_REPLY_POOL_SIZE=300
```

如果回复过度提起旧事，优先把 `MEMORY_REPLY_LIMIT` 调小，或把 `MEMORY_REPLY_MIN_SCORE` 调高。

回复时短期上下文也可以在根目录 `.env` 中配置：

```env
SHORT_CONTEXT_ENABLED=true
SHORT_CONTEXT_LIMIT=20
SHORT_CONTEXT_MAX_CHARS=1200
SHORT_CONTEXT_WINDOW_MINUTES=30
KAKA_OWNER_USER_IDS=你的QQ号
```

第一版短期上下文只取同场景最近输入，排除当前消息，默认只看最近 30 分钟，最多 20 条、总计 1200 字。如果卡咔接不上刚才的话，可以适当调大条数或时间窗口；如果被最近闲聊干扰，可以调小或关闭。

回复时关系上下文也会进入 prompt。第一版只做粗分，不做好感度分数：

```text
owner：QQ 号在 KAKA_OWNER_USER_IDS
familiar：历史输入 >=100，或最近 7 天输入 >=30，或 active 记忆 >=8
regular：历史输入 >=30，或最近 7 天输入 >=10，或 active 记忆 >=3
stranger：其他情况
```

如果卡咔对新人太熟，调高 `RELATIONSHIP_REGULAR_*` 阈值；如果一直把常见群友当陌生人，调低 regular 阈值。

自动候选分析可以在根目录 `.env` 中开启：

```env
MEMORY_AUTO_ANALYSIS_ENABLED=true
MEMORY_AUTO_ANALYSIS_TRIGGER_COUNT=50
MEMORY_AUTO_ANALYSIS_BATCH_LIMIT=50
MEMORY_AUTO_ANALYSIS_MAX_RUNS_PER_CHECK=2
```

开启后，`kaka-core` 启动时不会立刻分析；它会等到下一个整点检查，之后每个整点检查一次。只有 `not_analyzed >= 50` 时才调用大模型，每个整点最多跑两轮。

自动候选区复核可以在根目录 `.env` 中开启：

```env
MEMORY_AUTO_REVIEW_ENABLED=true
MEMORY_AUTO_REVIEW_TRIGGER_COUNT=20
MEMORY_AUTO_REVIEW_BATCH_SIZE=10
MEMORY_AUTO_REVIEW_MAX_RUNS_PER_CHECK=1
```

开启后，`kaka-core` 同样等到下一个整点检查。只有 `memory_candidates.pending >= 20` 时才调用大模型；每批 10 条，每个整点最多跑一批。复核通过会写入正式 `memories`，拒绝或重复会更新候选状态。

## 常见问题排查

### QQ 完全收不到消息

优先看 NapCat：

```text
1. QQ 是否在线。
2. NapCat 是否正在运行。
3. OneBot V11 是否启用。
4. 反向 WebSocket 地址是否是 ws://127.0.0.1:8081/onebot/v11/ws。
5. qq-adapter 是否已经启动并监听 8081。
```

如果 NapCat 日志里没有把事件推给 `127.0.0.1:8081`，问题通常在 NapCat 或 OneBot 配置。

### NapCat 连接不上 qq-adapter

检查：

```text
1. apps/qq-adapter/bot.py 是否正在运行。
2. 8081 是否被别的程序占用。
3. NapCat 配置里的 host、port、path 是否正确。
4. 地址是否写成 ws://127.0.0.1:8081/onebot/v11/ws。
```

如果改了 `qq-adapter` 端口，必须同步改 NapCat 的反向 WebSocket 地址。

### 私聊能回，群聊不回

先确认是不是触发规则导致的正常现象：

```text
群聊普通消息：不回复是正常的。
群聊包含“卡咔”：应该回复。
群聊 @卡咔：应该回复。
```

如果“卡咔”能触发但 @ 不能触发，重点检查 `qq-adapter` 的 @ 解析逻辑。

### 群聊普通消息不回复但数据库有记录

这是正常现象。

```text
群聊普通消息：写入 inputs，analysis_status=not_analyzed，没有 outputs。
群聊“卡咔”触发：写入 inputs 和 outputs，output_origin=passive，output_reason=keyword。
群聊 @触发：写入 inputs 和 outputs，output_origin=passive，output_reason=mention。
私聊文本：写入 inputs 和 outputs，output_origin=passive，output_reason=private。
```

如果想只看回复记录，可以用：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_recent_conversations.py --replied-only
```

如果想只看普通观察记录，可以用：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_recent_conversations.py --observed-only
```

### 群聊回复了但 outputs 没有记录

当前触发回复应该同时写入 `inputs` 和 `outputs`。

正常情况：

```text
群聊普通消息：不回复，只有 inputs。
群聊“卡咔”触发：记录。
群聊 @触发：记录。
私聊文本：记录。
```

如果已经回复但没有记录，优先检查 `kaka-core` 日志和 `DATABASE_URL`。

### 卡咔不回复，但 qq-adapter 有收到消息

分两种情况：

```text
群聊普通消息：正常，不会回复，但会观察记录。
触发消息无回复：检查 kaka-core 是否启动。
```

如果 `kaka-core` 没启动，QQ 侧应该返回：

```text
卡咔核心服务暂时没连上。
```

如果没有这个提示，说明问题可能在 `qq-adapter` 插件逻辑或消息没有进入处理器。

### kaka-core 启动失败

检查：

```text
1. 是否使用项目虚拟环境：
   D:\Python\AgentRobot\kaka-v2\.venv\Scripts\python.exe

2. 8001 是否被占用。

3. .env 是否存在必要的 LLM 配置。

4. DATABASE_URL 指向的 data 目录是否存在、是否可写。
```

不要把 `.env` 内容发到聊天里，也不要提交到 Git。

### LLM 没有真正调用

在 `kaka-core` 的 Swagger 页面测试：

```text
http://127.0.0.1:8001/docs
```

调用：

```text
POST /v1/chat
```

只观察不回复的接口是：

```text
POST /v1/observe
```

如果响应 metadata 里有 `llm_model`，说明调用到了模型。

如果响应里有 `fallback_reason`，说明模型调用失败或被降级，具体原因看 `fallback_reason` 和 `kaka-core` 日志。
