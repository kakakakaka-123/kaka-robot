# 2026-05-05 开发日志

## 今日重点

今天主要围绕 Web 管理台的正式记忆管理做收口。

目标不是扩大后台范围，而是把用户实际会用的正式记忆页补完整：

```text
编号顺序稳定
每页 50 条分页
正式记忆新增
正式记忆编辑
回复检索时能看到真实回复上下文
```

最近对话、输入分析和候选区仍保留在管理 API、脚本和数据库中，不再作为日常 Web 页面暴露。

## 正式记忆分页和排序

正式记忆列表后端改为按 `MemoryRecord.id.asc()` 排序，避免因为更新时间变化导致编号看起来乱跳。

`/admin/api/memories` 现在支持：

```text
limit
offset
total
```

前端默认每页 50 条，显示当前页、总页数、当前条目范围和总数，并提供上一页/下一页。

## 回复上下文预览

新增管理接口：

```text
POST /admin/api/reply-context/preview
```

它复用真实回复上下文组装逻辑，前端在“回复检索”页展示：

```text
System Prompt
User Prompt
metadata
used_memory_ids
memory_count
memory_injection_enabled
```

这个功能主要用于调试 prompt 和长期记忆注入，确认真实回复前到底给了模型什么上下文。

## 正式记忆新增和编辑

新增管理接口：

```text
POST /admin/api/memories
PATCH /admin/api/memories/{memory_id}
```

Web 正式记忆页新增：

```text
新增按钮
编辑按钮
内联编辑表单
```

新增记忆会写入：

```text
source = "manual"
merge_reason = "手动新增"
```

编辑支持修改：

```text
用户号
显示名
群号 / 私聊场景
memory_text
normalized_text
memory_type
confidence
source_text
status
merge_reason
```

归档、恢复和确认后硬删除保持原逻辑。

## 当前结论

正式记忆管理现在已经补齐基础 CRUD 中缺的“增”和“改”，再加上已有的归档、恢复、硬删除，Web 端已经能覆盖日常正式记忆维护。

下一步建议不要急着扩后台页面，先继续用真实 QQ 链路观察长期记忆是否自然参与回复；后续边加功能边调 prompt。
