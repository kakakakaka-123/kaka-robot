# 2026-05-05 开发日志

## 今日重点

今天主要围绕 Web 管理台的正式记忆管理做收口。

目标不是扩大后台范围，而是把用户实际会用的正式记忆页补完整：

```text
编号顺序稳定
每页 50 条分页
正式记忆新增
正式记忆编辑
回复检索时能看到真实回复上下文
```

最近对话、输入分析和候选区仍保留在管理 API、脚本和数据库中，不再作为日常 Web 页面暴露。

## 正式记忆分页和排序

正式记忆列表后端改为按 `MemoryRecord.id.asc()` 排序，避免因为更新时间变化导致编号看起来乱跳。

`/admin/api/memories` 现在支持：

```text
limit
offset
total
```

前端默认每页 50 条，显示当前页、总页数、当前条目范围和总数，并提供上一页/下一页。

## 回复上下文预览

新增管理接口：

```text
POST /admin/api/reply-context/preview
```

它复用真实回复上下文组装逻辑，前端在“回复检索”页展示：

```text
System Prompt
User Prompt
metadata
used_memory_ids
memory_count
memory_injection_enabled
```

这个功能主要用于调试 prompt 和长期记忆注入，确认真实回复前到底给了模型什么上下文。

## 正式记忆新增和编辑

归档、恢复和确认后硬删除保持原逻辑。

## 短期上下文第一版

随后补上回复前短期上下文。

新增配置：

```text
SHORT_CONTEXT_ENABLED=true
SHORT_CONTEXT_LIMIT=8
SHORT_CONTEXT_MAX_CHARS=1200
SHORT_CONTEXT_WINDOW_MINUTES=30
```

第一版规则：

```text
只读 inputs / outputs，不建新表
按同一 platform / scene_type / scene_id 取最近输入
排除当前 event_id
只看最近 30 分钟内的记录
默认最多 8 条输入记录
总字符上限 1200
普通观察消息和已有卡咔回复都可以进入上下文
```

上下文接入 `kaka_core.context.builder`，和长期记忆一起组成回复前 prompt。metadata 会记录：

```text
short_context_enabled
short_context_count
short_context_input_ids
short_context_limit
short_context_max_chars
```

本轮补充测试：

```text
同场景最近消息能进入 User Prompt
不同场景不会串
当前消息不会重复进入短期上下文
关闭 SHORT_CONTEXT_ENABLED 后不注入
/admin 回复上下文预览能看到短期上下文 metadata
```
