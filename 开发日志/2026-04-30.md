## 今日关键决策

### 1. 先做文本，不做图片和表情包

原因：

文本链路是第一阶段地基。图片、表情包、语音会引入文件下载、媒体存储、多模态模型、标签系统等复杂问题，不适合在地基阶段一起做。

### 2. 先做 LLMClient，再预留 LLMRouter

当前只使用 DeepSeek 作为普通聊天模型。

但结构上已经预留：

```text
普通聊天模型
复杂推理模型
记忆整理模型
工具决策模型
```

未来可以让不同任务调用不同模型，而不是把卡咔绑定死在一个模型上。

### 3. qq-adapter 先做内部流水线，不急着登录 QQ

原因：

真实 QQ 接入涉及 NoneBot、OneBot、NapCat 或 Lagrange。如果一开始就接，问题会混在一起。

先把适配器内部逻辑测通，后面接真实 QQ 会更稳。

### 4. 真实 QQ 接入使用 NoneBot2 + NapCat

本次实际选择：

```text
NoneBot2
-> nonebot-adapter-onebot
-> NapCat OneBot V11 反向 WebSocket
```

当前 `qq-adapter` 的 NoneBot 启动入口：

```text
apps/qq-adapter/bot.py
```

由于本机 `8080` 端口被旧的 Python 机器人进程占用，当前项目的 NoneBot 监听端口改为：

```text
127.0.0.1:8081
```

NapCat 的反向 WebSocket 地址配置为：

```text
ws://127.0.0.1:8081/onebot/v11/ws
```

实际登录并测试成功的 QQ 对应配置文件：

```text
NapCat.Shell.Windows.OneKey/.../resources/app/napcat/config/onebot11_1537786366.json
```

### 5. 群聊触发限制

真实 QQ 闭环跑通后，给群聊加了最小触发限制：

```text
私聊：全部回复
群聊：只有 @ 机器人 或 文本中包含“卡咔”时回复
```

这样可以避免卡咔在群里每句话都插嘴，也能减少不必要的 LLM 调用。

### 6. 先用 SQLite 记录原始对话

基础数据库记录已经接入 `kaka-core`，位置：

```text
services/kaka-core/src/kaka_core/storage/
```

当前新增：

```text
database.py      创建数据库连接和初始化表
models.py        定义 users / scenes / messages / responses 四张表
repository.py    保存一次 MessageEvent + KakaResponse
```

默认数据库文件：

```text
data/kaka.sqlite3
```

可以通过环境变量切换：

```text
DATABASE_URL=sqlite:///D:/Python/AgentRobot/kaka-v2/data/kaka.sqlite3
```

当前记录策略：

```text
收到 MessageEvent
-> 生成 KakaResponse
-> 保存用户、场景、输入消息、卡咔回复
-> 返回响应
```

数据库记录失败不会阻断聊天回复，只会在响应 metadata 中写入 `record_error`。

本地数据库文件已加入 `.gitignore`，避免误提交真实聊天记录。
