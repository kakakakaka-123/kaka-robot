## 长期记忆候选分析

围绕 `inputs.analysis_status=not_analyzed`，继续完善最小输入分析工具：

```text
services/kaka-core/scripts/analyze_inputs.py
```

当前定位：

- 规则先做保守初筛。
- 明显低价值内容标记为 `skipped`。
- 身份、偏好、项目长期信息等标记为 `candidate`。
- 上下文不足或普通聊天内容保留为 `not_sure`。
- `--llm-batch` 会把 `not_sure` 按场景和时间打包交给大模型判断。
- `--write-candidates` 才会写入 `memory_candidates` 并更新来源输入状态。
- `--mark-skipped` 才会把 skipped 写回数据库。

批量判断策略已经调整为：

```text
每包最多 10 条待判断 not_sure
同一包时间跨度最多 10 分钟
前后各带 5 条上下文
上下文也限制在 10 分钟窗口内
```

这样做的目的：

- 减少单条请求成本。
- 给模型一点上下文。
- 避免跨很久的消息被错误拼在一起。
- 保持当前阶段只打印或写候选区，不直接污染正式长期记忆。

## 候选区和正式记忆表

当前数据库长期记忆相关表：

```text
memory_candidates
memories
```

`memory_candidates` 用作候选区，只接收分析脚本或自动分析产生的候选。

`memories` 是正式长期记忆表，当前由合并脚本或测试阶段的 LLM 复核脚本写入；不让原始分析模型直接绕过候选区写入。

新增正式记忆模型和建表/迁移检查：

```text
services/kaka-core/src/kaka_core/storage/models.py
services/kaka-core/src/kaka_core/storage/database.py
```

新增候选合并脚本：

```text
services/kaka-core/scripts/merge_memory_candidates.py
```

合并脚本当前行为：

- 默认只读预览，不修改数据库。
- 加 `--apply` 后才写入 `memories`。
- 成功合并的候选标记为 `approved`。
- 与已有正式记忆重复的候选标记为 `merged_duplicate`。
- 基础去重规则为同一 `user_id`、同一 `memory_type`、同一归一化正文。

当前推荐使用方式：

```powershell
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\show_memory_candidates.py
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\merge_memory_candidates.py --limit 20
.\.venv\Scripts\python.exe .\services\kaka-core\scripts\merge_memory_candidates.py --limit 20 --apply
```

第一条查看候选区，第二条只读预览合并计划，第三条小批量写入正式 `memories`。

新增正式记忆查看脚本：

```text
services/kaka-core/scripts/show_memories.py
```

用途：

- 只读查看 `memories`，不修改数据库。
- 支持按状态、类型、来源、群、用户、日期筛选。
- 显示正式记忆正文 `memory_text` 和去重文本 `normalized_text`。
- 显示来源候选、来源消息和合并理由，方便人工检查正式记忆质量。

新增正式记忆检索预览脚本：

```text
services/kaka-core/scripts/search_memories.py
```

用途：

- 只读读取 `memories`，不修改数据库。
- 按当前用户和当前消息检索少量 `active` 正式记忆。
- 可选传入群号或私聊场景，用于同场景加分。
- 打印检索分数、命中词和命中原因。
- 当前脚本已复用正式检索模块，聊天回复前也使用同一套检索逻辑。

当前检索规则保持保守：

```text
硬条件：同一 user_id + status=active
默认候选池：该用户最近 300 条 active 记忆
默认返回：最多 5 条
主要加分：关键词命中、同场景、记忆类型、置信度、近期更新
```

新增回复上下文组装器：

## 长期记忆 E2E 回放

本轮新增：

```text
services/kaka-core/scripts/review_memory_candidates.py
services/kaka-core/scripts/seed_memory_e2e_data.py
services/kaka-core/tests/test_review_memory_candidates.py
```

`review_memory_candidates.py` 用于测试阶段临时代替人工审核候选区：

- 默认调用大模型只读预览。
- 加 `--apply` 后把 approve 写入正式 `memories`。
- reject 会把候选标记为 `rejected`。
- duplicate 会把候选标记为 `merged_duplicate`。
- 支持 LLM 解析失败或空返回时自动拆包重试。
- 增加“他是我/她是我/这是我...”关系事实兜底。
- 增加偏好类正式记忆类型归一化，避免“回复偏好”被写成 `relationship_fact`。

E2E 回放链路：

```text
show_recent_conversations.py
-> analyze_inputs.py --llm-batch --write-candidates
-> show_memory_candidates.py
-> review_memory_candidates.py --apply
-> show_memories.py
-> search_memories.py
```

最终结果：

```text
测试输入：40 条
输入状态：analyzed 28 / skipped 12
候选状态：approved 23 / merged_duplicate 2 / rejected 3
正式记忆：23 条来自本轮 E2E 候选
```

本轮根据回放结果做的优化：

- `analyze_inputs.py` 的规则候选类型推断改为结合 reason 和原始文本，避免 `关系或协作要求` 把回复偏好误分为 `relationship_fact`。
- `review_memory_candidates.py` 写入正式记忆时同步候选类型，候选区查看和正式记忆查看保持一致。
- `review_memory_candidates.py` 对明确第一人称关系句增加确定性兜底，例如“他是我室友小陈，负责前端。”会写成“测试用户A的室友小陈负责前端。”。
- `review_memory_candidates.py` 对偏好类文本强制归一到 `stable_preference`。
