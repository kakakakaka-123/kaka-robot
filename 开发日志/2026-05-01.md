用法示例：


或：

```python

```

这样用户不需要在 PyCharm 里寻找 `Parameters` / `Script parameters`，直接改 `.py` 顶部配置后右键运行即可。

规则：

## @ 触发和端口残留修复

用户测试发现：

- 群聊普通消息不触发：正常。
- 群聊包含“卡咔”触发：正常。
- 群聊 @ 机器人触发：异常。

排查结果：

1. @ 触发异常的原因是插件只用 `event.get_plaintext()` 作为最终消息文本，@ 消息里的 at 段需要单独判断，同时发给 `kaka-core` 时应跳过 at 段，只保留 text 段。
2. 关闭 `run.py` 后仍能运行，是因为本机同时存在系统 Python 和 `.venv` Python 启动的重复 `bot.py` / `run.py` 进程，且 `8000` 出现了无法通过进程查询清理的残留监听。

已修复：

- 拆出 `_is_at_bot` 判断 @ 当前机器人。
- 拆出 `_extract_text_for_core`，@ 消息只把 text 段发给核心服务。
- 新增 @ 消息测试，并兼容 NapCat 日志中出现的 `[at:qq=...]` 形式。
- 修复 `kaka-core` 关闭时 `httpx` 连接异常未被包装的问题，现在会提示“卡咔核心服务暂时没连上”。
- 停止重复的当前项目 Python 服务。
- 将本地开发 `kaka-core` 端口从 `8000` 改为 `8001`，避免残留端口影响。
- 将 `qq-adapter` 默认 `KAKA_CORE_BASE_URL` 从 `http://127.0.0.1:8000` 改为 `http://127.0.0.1:8001`。

## @ 空消息触发修复

```text

```

这样核心服务可以正常生成回复，不会因为空文本被丢弃。

## messages / responses 表语义调整

这次根据后续用户画像、情绪识别和长期记忆的需要，调整了数据库记录设计。

核心结论：

```text
messages = 卡咔看见的消息
responses = 卡咔实际发出的回复
```

也就是说：

```text
普通群聊文本
-> 不回复
-> 不调用 DeepSeek
-> 写入 messages
-> 不写入 responses

私聊 / @卡咔 / 包含“卡咔”
-> 调用 /v1/chat
-> 回复
-> 写入 messages
-> 写入 responses
```

这比“只记录触发消息”更适合后续做用户画像和上下文理解，因为群聊里不可能要求每个人每句话都 @ 卡咔或带关键词。

### 表结构变化

`messages` 新增字段：

```text
is_processed      是否进入处理流程
process_reason    处理原因，例如 none / private / mention / keyword
analysis_status   后续分析状态，例如 pending / done / skipped / failed
```

当前普通观察消息默认：

```text
is_processed=false
process_reason=none
analysis_status=pending
```

触发回复消息默认：

```text
is_processed=true
process_reason=private / mention / keyword
analysis_status=done
```

`responses.should_reply` 暂时保留。当前普通观察消息不会写入 `responses`，但未来如果出现“已处理但策略决定不回复”的情况，这个字段仍有价值。

### 接口变化

`kaka-core` 新增：

```text
POST /v1/observe
```

作用：

```text
只记录消息
不调用 LLM
不回复
返回 should_reply=false
```

`POST /v1/chat` 继续负责触发回复。

### 适配器变化

`qq-adapter` 的群聊处理从“触发或丢弃”改成：

```text
私聊：/v1/chat，process_reason=private
群聊 @机器人：/v1/chat，process_reason=mention
群聊包含“卡咔”：/v1/chat，process_reason=keyword
群聊普通文本：/v1/observe，process_reason=none
```

同时 QQ 事件现在会用 QQ 场景和 `message_id` 生成稳定 `event_id`，避免同一条消息重复写入。

### 数据库迁移

当前项目还没有引入 Alembic，因此本次只做 SQLite 轻量补列：

```text

```

不会删除或重建数据库，不会清空已有真实聊天记录。旧记录补列后默认视为：

```text
is_processed=true
process_reason=legacy
analysis_status=done
```

## 表名最终调整为 inputs / outputs

继续讨论后发现，`messages / responses` 这组名称不够贴合长期语义：


因此最终将表名调整为：

```text

```

当前长期语义：


