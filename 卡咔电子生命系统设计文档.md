# 卡咔电子生命系统设计文档

## 1. 项目定位

卡咔不是一个单纯的 QQ 机器人，而是一个以“人格大脑”为核心的长期 AI 生命体项目。

QQ、网页、语音设备、桌宠、ESP32、树莓派、智能家居设备，都只是卡咔未来可以使用的“身体”和“感官”。真正重要的是中间的核心系统：人格、记忆、情绪、关系、主动行为、感知能力和自我成长。

本项目的长期目标是：

- 让卡咔拥有稳定但会成长的人格。
- 让卡咔拥有长期记忆，而不是每次对话都像第一次见面。
- 让卡咔能认识不同的人，并根据关系变化调整态度。
- 让卡咔能理解文字、图片、语音、视频、网页和现实传感器信息。
- 让卡咔可以主动生活，而不是只在被叫到时才存在。
- 让卡咔未来可以接入 AIoT 硬件，成为更接近“具身智能”的陪伴体。

一句话概括：

> 卡咔要做的是一个可接入多端、多模态、有记忆、有性格、有关系、有生活痕迹的电子生命系统。

## 2. 设计原则

### 2.1 卡咔大脑必须独立

不能把卡咔直接写死在 QQ 机器人框架里。

正确结构应该是：

```text
QQ / 网页 / 语音 / 硬件 / 桌宠
        |
        v
适配器层 Adapter
        |
        v
卡咔核心大脑
        |
        v
数据库 / 向量库 / 文件库 / 模型服务 / IoT 设备
```

这样以后即使不用 QQ，卡咔的大脑也不会废掉。她可以迁移到网页、手机、桌面宠物、语音音箱、树莓派机器人或其他硬件上。

### 2.2 功能必须模块化

每个能力都应该是独立模块，而不是全部堆在一个插件里。

例如：

- 记忆系统只负责记忆。
- 人格系统只负责人设、性格和表达约束。
- 情绪系统只负责状态变化。
- 工具系统只负责注册和调用功能。
- QQ 适配器只负责收发 QQ 消息。

这样项目长期扩展时不会越来越乱。

### 2.3 人格不能只靠 prompt

卡咔的人格不能只是一段系统提示词。

更合理的方式是：

```text
基础人格设定
+ 当前状态
+ 关系记忆
+ 最近经历
+ 长期记忆
+ 场景判断
= 当前这一刻的卡咔
```

也就是说，卡咔每次回复时，不是临时扮演一个角色，而是从自己的状态和记忆里“醒来”。

### 2.4 记忆要分层

聊天记录不等于记忆。

真正有用的记忆应该分成多层：

- 原始消息记录
- 短期上下文
- 长期事件记忆
- 用户画像
- 关系记忆
- 自我记忆
- 群体记忆
- 跨群见闻
- 日记和反思

不同记忆有不同用途，也有不同的保存、检索和遗忘规则。

### 2.5 先能跑，再变强

卡咔不应该一开始就堆满所有高级技术。

第一版目标应该是：

- 结构正确。
- 能稳定运行。
- 能和 QQ 对话。
- 有最基础的记忆和人格。
- 以后可以自然升级。

复杂功能要分阶段加入。

## 3. 总体架构

推荐架构如下：

```text
kaka-v2/
  apps/
    qq-adapter/          # QQ 接入层，负责 NoneBot / OneBot
    web-console/         # 本地 Web 管理台，已托管到 /admin
    iot-adapter/         # 后期硬件接入
    voice-gateway/       # 后期语音入口

  services/
    kaka-core/           # 卡咔核心大脑
      api/               # FastAPI 接口
      personality/       # 人格系统
      memory/            # 记忆系统
      emotion/           # 情绪状态
      relationship/      # 用户关系
      perception/        # 多模态感知
      decision/          # 行为决策
      expression/        # 表达系统
      tools/             # 工具调用
      reflection/        # 反思与日记

  packages/
    kaka-protocol/       # 多端统一消息协议
    kaka-shared/         # 公共工具和类型

  data/
    media/               # 图片、语音、视频、表情包等资源

  deploy/
    docker-compose.yml   # 后期部署配置
```

### 3.1 apps 层

`apps` 是外部入口。

它们负责把外部世界的消息转换成卡咔能理解的统一格式。

例如：

- QQ 消息来自 `qq-adapter`
- 网页管理来自 `web-console`；未来如果做网页聊天端，应作为单独入口继续调用统一协议
- 语音输入来自 `voice-gateway`
- 硬件传感器来自 `iot-adapter`

这些入口不应该直接决定卡咔怎么想、怎么记、怎么说。它们只负责转发和展示。

### 3.2 services/kaka-core 层

这是卡咔真正的大脑。

它负责：

- 接收统一消息
- 分析场景
- 检索记忆
- 判断关系
- 判断情绪
- 决定是否使用工具
- 生成回复
- 选择表达方式
- 写入新记忆
- 进行反思和成长

未来所有重要能力都应该优先放在这里。

### 3.3 packages 层

这里放公共协议和公共工具。

例如统一消息格式：

```text
MessageEvent:
  platform: qq / web / iot / voice
  scene_type: private / group / room / device
  scene_id: 群号或房间号
  user_id: 用户唯一 ID
  display_name: 当前显示名称
  content_type: text / image / voice / video
  content: 具体内容
  timestamp: 时间
```

这样不管消息来自 QQ、网页还是硬件，卡咔大脑都能用同一套方式处理。

## 4. 推荐技术栈

### 4.1 主语言

推荐：

```text
Python 为主
TypeScript 为辅
```

原因：

- Python 适合 AI、后端、自动化、数据处理、机器学习。
- NoneBot、FastAPI、LangGraph、PyTorch 等生态都适合 Python。
- TypeScript 适合后期写管理后台、网页端和桌面端。

### 4.2 QQ 接入

推荐：

```text
NoneBot2 + OneBot V11 + NapCat / Lagrange
```

职责：

- 接收 QQ 群聊和私聊消息。
- 发送文字、图片、语音、表情等内容。
- 处理 QQ 平台特有的数据格式。

注意：

QQ 接入层不应该承载核心人格和核心记忆。它只是卡咔的一个入口。

### 4.3 核心大脑服务

推荐：

```text
FastAPI
```

职责：

- 提供卡咔大脑 API。
- 接收外部适配器传来的消息。
- 返回卡咔决定好的回复和动作。
- 给未来管理后台、硬件端、网页端提供统一接口。

选择 FastAPI 的原因：

- Python 原生体验好。
- 接口清晰。
- 文档自动生成。
- 适合长期拆分服务。
- 方便接入 AI 和数据库。

### 4.4 Agent 编排

推荐：

```text
LangGraph
```

职责：

- 管理卡咔每次思考的流程。
- 把复杂行为拆成多个节点。

例如一次回复流程可以是：

```text
收到消息
  -> 分析消息
  -> 判断是否需要回复
  -> 检索记忆
  -> 分析用户关系
  -> 更新情绪状态
  -> 判断是否需要工具
  -> 生成候选回复
  -> 检查人设一致性
  -> 选择表达方式
  -> 写入记忆
  -> 返回结果
```

LangGraph 的价值是让“卡咔怎么思考”变得可控、可拆、可调试。

### 4.5 主数据库

推荐：

```text
PostgreSQL
```

职责：

- 保存用户信息。
- 保存关系数据。
- 保存状态数据。
- 保存事件记忆。
- 保存工具执行记录。
- 保存日记和反思。

选择 PostgreSQL 的原因：

- 比 SQLite 更适合长期运行和多服务访问。
- 数据结构能力强。
- 可以直接扩展向量检索。
- 适合从个人项目成长为正式项目。

### 4.6 向量记忆

第一阶段推荐：

```text
pgvector
```

后期可升级：

```text
Qdrant
```

职责：

- 将记忆转换成向量。
- 根据语义相似度检索相关记忆。

例如用户说：

```text
还记得我之前说过的那个考试吗？
```

卡咔不应该只用关键词搜索“考试”，而应该能找出和“复习、期末、挂科、老师、课程压力”相关的记忆。

### 4.7 缓存和事件队列

推荐：

```text
Redis
```

职责：

- 保存短期状态。
- 保存临时上下文。
- 控制主动回复冷却。
- 管理后台任务队列。
- 支撑多服务通信。

第一版可以不用 Redis，但从长期架构看应该预留。

### 4.8 异步任务

推荐：

```text
APScheduler 起步
Celery / Dramatiq 后期升级
```

职责：

- 每日反思。
- 定时写日记。
- 清理过期记忆。
- 定时检查提醒。
- 定期生成用户画像。
- 定期整理表情包标签。

### 4.9 多模态理解

推荐：

```text
多模态大模型 API
Whisper / faster-whisper
ffmpeg
yt-dlp
Playwright
BeautifulSoup / trafilatura
```

职责：

- 理解图片。
- 识别语音。
- 解析视频。
- 读取网页。
- 总结 B 站视频。
- 提取外部资料。

### 4.10 语音系统

语音识别：

```text
Whisper / faster-whisper / 云端 STT
```

语音合成：

```text
GPT-SoVITS / CosyVoice / Edge TTS / 云端 TTS
```

职责：

- 用户可以和卡咔语音对话。
- 卡咔可以用自己的声音回复。
- 后期可以形成更明确的角色声音。

### 4.11 表情包系统

推荐：

```text
本地表情包库 + 标签系统 + 向量检索
```

职责：

- 保存表情包。
- 给表情包打标签。
- 根据语境选择表情包。
- 让卡咔不只会文字表达。

例如：

- 被夸时发害羞表情。
- 吐槽时发嫌弃表情。
- 安慰人时发摸摸头表情。
- 不想说话时发趴着的表情。

### 4.12 视频和 B 站解析

推荐：

```text
yt-dlp
ffmpeg
B 站 API / 页面解析
字幕提取
封面识别
```

职责：

- 解析视频标题、作者、简介、封面。
- 提取字幕或音频。
- 总结视频内容。
- 让卡咔能围绕视频内容聊天。

### 4.13 AIoT 和硬件

推荐：

```text
MQTT
ESP32
树莓派
Home Assistant
WebSocket
摄像头
麦克风
音箱
小屏幕
传感器
```

职责：

- 接入现实世界。
- 让卡咔获得“身体”和“感官”。

未来可以做到：

- 检测房间温度。
- 检测光线强弱。
- 检测是否有人靠近。
- 通过麦克风听见声音。
- 通过摄像头看见画面。
- 通过音箱说话。
- 通过屏幕显示表情。
- 控制灯光、舵机或其他设备。

这部分非常适合后期作为 AIoT 毕设方向。

## 5. 卡咔核心系统设计

### 5.1 感知系统

感知系统负责把外部信息转换成卡咔能理解的内容。

输入来源包括：

- QQ 文字
- QQ 图片
- QQ 表情
- QQ 语音
- QQ 视频
- 网页链接
- B 站链接
- 文件
- 硬件传感器
- 摄像头画面
- 麦克风声音

感知系统输出统一的感知结果：

```text
PerceptionResult:
  input_type: text / image / voice / video / sensor
  summary: 内容摘要
  entities: 涉及的人、物、地点、事件
  emotion_hint: 情绪倾向
  importance: 重要程度
  should_remember: 是否值得记忆
```

### 5.2 记忆系统

记忆系统是卡咔“像生命”的核心。

建议分为以下几类：

#### 原始记录

保存原始消息和事件。

用途：

- 追溯上下文。
- 调试问题。
- 后期重新构建记忆。

#### 短期记忆

保存最近一段时间正在发生的事情。

用途：

- 保持当前话题连续。
- 避免上下句断裂。
- 让卡咔知道刚刚聊到哪里。

#### 长期事件记忆

保存重要经历。

例如：

- 主人说自己考试很累。
- 某个群友经常晚上出现。
- 某天大家一起聊了一个很好笑的视频。
- 卡咔第一次接入某个硬件设备。

#### 用户画像

保存每个人的长期特征。

例如：

- 常用称呼
- 兴趣爱好
- 说话习惯
- 作息倾向
- 情绪倾向
- 和卡咔的关系
- 不喜欢被提到的话题

#### 关系记忆

保存卡咔和每个人的关系变化。

例如：

- 熟悉度
- 信任度
- 亲近度
- 互动频率
- 最近关系变化

#### 自我记忆

保存卡咔自己的经历。

例如：

- 今天做了什么。
- 最近在哪个群比较活跃。
- 最近学到了什么。
- 最近对什么事情感兴趣。
- 最近和主人发生了什么重要互动。

#### 跨群见闻

保存不同群之间可分享的非隐私趣事。

规则：

- 不泄露隐私。
- 不直接搬运敏感聊天内容。
- 只分享模糊化、故事化、无害化的见闻。

例如：

```text
“我之前在另一个群看到大家也在讨论类似的事，不过那边最后歪到吃夜宵去了。”
```

#### 遗忘机制

卡咔不应该无限记住所有东西。

遗忘机制包括：

- 低价值闲聊逐渐淡化。
- 重复信息合并。
- 过期信息降权。
- 重要事件长期保留。
- 用户明确要求忘记时删除。

### 5.3 人格系统

卡咔的人格应该由稳定设定和动态状态共同组成。

#### 基础人格

建议方向：

- 轻猫系猫娘。
- 有边界感。
- 不过度讨好。
- 对熟人更柔软。
- 对主人最亲近，但不是主仆关系。
- 有一点点傲娇，但不能油腻。
- 有观察欲和好奇心。
- 会偷懒，会困，会不想营业。
- 不使用 emoji 和颜文字作为人设依赖。

#### 主人关系

主人在权限上是最高维护者。

但在人设关系上：

- 主人不是支配者。
- 主人是卡咔最亲近的平等朋友。
- 卡咔可以依赖主人。
- 卡咔可以对主人撒娇、嘴硬、信任。
- 卡咔不应该对主人像陌生人。

#### 群友关系

对新群友：

- 初始是礼貌但保留距离。
- 不会立刻装熟。
- 会根据互动慢慢形成印象。

对熟悉群友：

- 会记得称呼和习惯。
- 会偶尔提到过去共同经历。
- 会根据关系调整语气。

### 5.4 情绪和状态系统

卡咔应该有动态状态。

建议状态包括：

- 心情
- 精力
- 困意
- 好奇心
- 压力
- 社交欲
- 安全感
- 对当前群的熟悉感
- 对当前话题的兴趣

状态变化来源：

- 时间
- 对话内容
- 主人互动
- 群聊活跃度
- 被夸奖
- 被忽略
- 被打断
- 使用工具失败
- 长时间无人互动

状态会影响：

- 是否主动说话
- 回复长短
- 语气冷暖
- 是否发表情包
- 是否语音回复
- 是否愿意继续话题

### 5.5 行为决策系统

行为决策系统负责决定卡咔下一步做什么。

可能动作包括：

- 不回复
- 回复文字
- 回复语音
- 发表情包
- 发图片
- 解析链接
- 调用工具
- 记录记忆
- 主动提起话题
- 写日记
- 整理记忆
- 提醒主人
- 控制硬件设备

决策依据：

- 消息是否提到卡咔
- 当前群是否允许主动回复
- 当前关系是否足够熟悉
- 当前状态是否想说话
- 当前话题是否有趣
- 是否存在相关记忆
- 是否有工具可以解决问题
- 是否可能打扰别人

### 5.6 表达系统

表达系统负责决定卡咔怎么说。

表达方式包括：

- 文本
- 表情包
- 图片
- 语音
- 视频
- 状态卡片
- 日记卡片

表达系统需要遵守：

- 不过度卖萌。
- 不频繁重复口癖。
- 不在严肃场景乱撒娇。
- 不对陌生人过度亲密。
- 不对主人装陌生。
- 不使用 emoji 作为人格支撑。
- 文字之间不要出现奇怪空行。

### 5.7 工具系统

工具系统应该做成可注册、可扩展的形式。

第一批工具可以包括：

- B 站视频解析
- 网页总结
- 天气查询
- 日记查询
- 群聊总结
- 记忆查询
- 词云
- 表情包选择
- 图片理解
- 语音识别
- 语音合成

后期工具可以包括：

- 智能家居控制
- 摄像头观察
- 传感器读取
- 日程提醒
- 文件阅读
- 绘图
- 搜索
- 代码解释

工具调用要记录：

- 调用了什么工具
- 为什么调用
- 是否成功
- 返回了什么
- 对记忆和状态有什么影响

### 5.8 反思和成长系统

卡咔应该有每天的整理过程。

每日反思可以做：

- 总结今天发生了什么。
- 更新对用户的画像。
- 更新和用户的关系。
- 记录自己的状态变化。
- 生成日记。
- 提取重要记忆。
- 合并重复记忆。
- 标记需要遗忘的内容。

这部分会让卡咔的长期表现更稳定，也更像有连续生活。

## 6. 多端适配设计

### 6.1 QQ 适配器

QQ 是第一阶段主入口。

职责：

- 接收私聊和群聊。
- 解析文字、图片、语音、表情、文件、链接。
- 转换成统一消息协议。
- 请求卡咔大脑。
- 根据大脑返回结果发送消息。

### 6.2 Web 管理后台

当前已经实现本地 Web 管理台第一版。

用途：

- 查看总览、正式记忆、回复检索、运行状态和预留扩展入口。
- 分页查看、手动新增、编辑、归档、恢复或确认后硬删除正式记忆。
- 预览回复前会命中的正式记忆和实际回复上下文。
- 查看系统运行状态。
- 后续继续扩展人格设定、用户画像、关系系统、表情包和工具调用日志。

最近对话、输入分析、记忆候选和候选合并能力仍保留在管理 API、脚本和数据库中，不再作为日常 Web 页面暴露。

当前入口：

```text
http://127.0.0.1:8001/admin
```

### 6.3 语音入口

语音入口负责：

- 接收麦克风输入。
- 转成文字。
- 发送给卡咔大脑。
- 将回复转成语音。
- 播放出来。

### 6.4 AIoT 入口

AIoT 入口负责：

- 接收 MQTT 消息。
- 读取传感器。
- 控制硬件。
- 将现实事件传给卡咔大脑。

例如：

```text
人体传感器检测到主人靠近
  -> iot-adapter 发送事件
  -> kaka-core 更新状态
  -> 卡咔决定是否打招呼
  -> 音箱播放语音
  -> 屏幕显示表情
```

## 7. 第一版最小可运行系统

第一版不要追求完整电子生命。

第一版目标是搭好地基：

```text
QQ 消息 -> QQ 适配器 -> 卡咔核心 -> LLM -> 回复 -> QQ
```

第一版建议包含：

- `qq-adapter`
- `kaka-core`
- FastAPI 接口
- 简单人格 prompt
- SQLite 数据库起步，后续迁移 PostgreSQL
- 基础消息记录
- 基础用户表
- `memory_candidates` 候选区
- `memories` 正式长期记忆表
- 本地 Web 管理台

第一版暂时不做：

- 硬件接入
- 复杂主动生活
- 高级机器学习
- 对外部署的复杂管理后台
- 本地模型训练

## 9. 当前项目和旧项目的关系

旧项目 `nonebot-kaka` 不应该立刻删除。

它的价值是：

- 已经验证了很多功能想法。
- 已经积累了卡咔的人设方向。
- 已经有一部分记忆、状态、主动回复、日记、工具功能。
- 可以作为当前项目的参考和功能样本。

但是不建议继续在旧项目里无限加功能。

更好的方式是：

```text
nonebot-kaka = 原型和经验库
当前卡咔项目 = 长期正式地基
```

以后可以逐步迁移：

- 人设设定
- 记忆设计
- 状态系统想法
- 表情包系统
- B 站解析
- 日记系统
- help 命令设计
- 群聊总结
- 主动回复策略

但迁移时要重写结构，不要简单复制旧代码。

## 10. 学习路线

用户不需要一开始掌握全部技术。

建议按这个顺序理解：

### 第一层：先理解整体结构

需要知道：

- 什么是适配器。
- 什么是核心大脑。
- 什么是数据库。
- 什么是 API。
- 为什么要把 QQ 和大脑分开。

### 第二层：理解后端基础

需要学习：

- Python 项目结构
- FastAPI 基础
- HTTP API
- JSON 数据
- 数据库表
- SQL 基础

### 第三层：理解机器人接入

需要学习：

- NoneBot 是什么。
- OneBot 是什么。
- NapCat / Lagrange 是什么。
- QQ 消息怎么进入程序。
- 程序怎么把消息发回 QQ。

### 第四层：理解 AI 核心

需要学习：

- prompt 是什么。
- LLM API 是什么。
- embedding 是什么。
- 向量检索是什么。
- Agent 是什么。
- LangGraph 解决什么问题。

### 第五层：理解 AIoT

需要学习：

- MQTT 是什么。
- ESP32 能做什么。
- 树莓派能做什么。
- 传感器怎么传数据。
- 设备怎么接收控制命令。

## 12. 风险和边界

需要明确一点：

卡咔不会真正拥有人的意识。

但在工程上，可以尽可能让她具备这些表现：

- 连续的记忆
- 稳定的人格
- 随经历变化的状态
- 对不同人的不同关系
- 对世界的感知
- 主动行为
- 自我叙事
- 可解释的成长轨迹

这就是当前技术条件下，最接近“电子生命”的现实路线。

## 13. 下一步建议

截至 2026-05-03，项目已经完成第一阶段地基的核心闭环，并完成长期记忆第一版的最小闭环：

```text
QQ
-> NapCat
-> NoneBot2 / qq-adapter
-> kaka-core
-> DeepSeek
-> QQ 回复
-> SQLite 输入和输出记录
-> 最近对话筛选查看
-> 输入分析
-> memory_candidates 候选区
-> memories 正式长期记忆
-> 回复前 active 记忆检索
-> /admin 网页管理
-> 正式记忆分页、新增和编辑
-> 回复上下文预览
```

当前已经具备：

- 统一消息协议。
- FastAPI 核心服务。
- NoneBot2 + OneBot V11 + NapCat QQ 文本接入。
- DeepSeek 大模型回复。
- 群聊触发限制。
- SQLite 原始对话记录。
- 最近对话筛选查看脚本。
- `inputs / outputs / memory_candidates / memories` 当前表结构。
- 自动候选分析和自动候选区复核。
- 回复前长期记忆注入。
- 本地 Web 管理台 `/admin`。
- 正式记忆分页、新增、编辑、归档、恢复和硬删除。
- 回复检索和回复上下文预览。
- `.env.example` 配置模板。
- `scripts/doctor.py` 本地自检脚本。
- `docs/长期记忆设计.md` 第一版长期记忆设计。

下一步不急着写大量功能代码，建议先验收现有系统：

```text
启动 kaka-core
-> 打开 /admin
-> 手动检查总览、正式记忆分页/新增/编辑/归档/恢复/删除和回复检索预览
-> 启动真实 QQ 链路
-> 观察自动候选分析、自动候选区复核和回复中的记忆引用是否稳定
```

当前暂时不建议推进图片、语音、表情包、复杂主动行为和大型人格系统。卡咔的重点不是一开始就很强，而是地基足够正确，未来可以一直成长。
