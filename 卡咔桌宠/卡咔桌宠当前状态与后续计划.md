# 卡咔桌宠当前状态与后续计划

更新时间：2026-05-30

## 定位

卡咔桌宠属于“卡咔 IP”部分，负责卡咔的桌面身体、状态表现、互动手感、视觉素材承载和后续动画资产落地。

它不是“卡咔机器人”核心本身。`kaka-core`、QQ 接入、长期记忆、Web 管理台、协议和真实模型回复链路属于“卡咔机器人”部分。

桌宠可以调用 `kaka-core /v1/chat`，但这只是把机器人核心作为本地大脑服务使用；桌宠路线不承接机器人核心、记忆系统或 QQ 适配器的开发任务。

主要分工：

- React / TypeScript：桌宠状态、右键菜单、设置面板、输入框、气泡、交互状态调度。
- PixiJS：透明 PNG 渲染、轻量动作、状态切换缓动。
- `kaka-core`：仅作为可选本地核心大脑，被桌宠通过 `/health` 和 `/v1/chat` 调用。

## 当前已完成

- 透明、无边框、置顶且不占任务栏的桌面窗口。
- 系统托盘：显示/隐藏、设置、开机自启、重置位置、修复窗口、连接测试、重启桌宠、退出。
- 设置窗口：开机自启、启动后是否显示卡咔、桌宠大小、气泡停留时间、随机待机频率、闲置睡觉时间、始终置顶、状态标签、右键调试菜单。
- 右键菜单：默认只保留说话、摸摸头、设置、隐藏、退出；调试状态菜单默认收起。
- 拖拽移动：支持窗口位置记忆、拖拽反应、长拖拽反馈、松手恢复。
- 摸头 / 双击 / 长按 / 睡醒 / 睡觉 / 随机待机反馈。
- 状态来源优先级：拖拽、对话、系统检查、回复播放、普通互动、睡觉和随机待机不会随意互相覆盖。
- 最小对话入口：右键或双击打开输入框，调用 `kaka-core /v1/chat` 后把回复拆成气泡显示。
- 对话失败处理：核心未启动或请求异常时显示信号弱，并保留输入方便重试。
- 最近对话临时回看：只保留本次运行内最近 6 条，不写数据库。
- 设置窗口连接测试会同步驱动桌宠本体状态。
- 对话回复气泡播放已修复：最近对话有回复时，卡咔本体也会显示气泡。

## 当前不做

这些不是当前桌宠线的必要项，暂不推进：

- 完整聊天历史窗口：用户不会高频使用桌宠聊天，收益不高。
- 全局快捷键 / 快捷唤醒：当前使用场景不强，暂不增加复杂度。
- 语音输入 / 语音播放：需要另行设计体验和资源，不适合在当前素材阶段硬加。
- 更多机器人核心能力：应归入“卡咔机器人”线，而不是桌宠 IP 线。

## 已知限制

- 当前是静态 PNG 加程序化轻量动作，不是逐帧动画、Live2D 或 Spine。
- 桌宠对话只做轻量气泡回复，不承担完整聊天客户端职责。
- 开机自启真实重启回归测试还需要用户方便重启电脑时再确认。
- Windows 11 隐藏托盘右键菜单自动化点选不稳定，`修复窗口` / `重启桌宠` 仍建议用户手动最终确认。
- `cargo fmt --check` 和 `cargo clippy` 本轮未执行，因为当前 Rust toolchain 未安装 `rustfmt` / `clippy` 组件。

## 后续怎么做

在没有新素材前，桌宠本体建议只做稳定性观察，不再继续堆功能。

新素材到位后，优先按下面顺序推进：

1. 素材盘点和命名规范

确认每个新素材属于哪类：

- 表情差分
- 姿态差分
- 动作序列
- 逐帧动画
- Live2D / Spine 原始工程

先统一文件名、状态名、用途和尺寸标准，再改代码。

2. 状态映射升级

把 `petStates.ts` 从“单状态单 PNG”升级成更清晰的素材映射：

- 基础状态
- 过渡状态
- 互动动作
- 一次性动作
- 循环动作

3. 渲染方案选择

根据素材形态决定方案：

- 仍是 PNG 差分：继续 PixiJS，补更细的状态切换和动作队列。
- 是逐帧序列：用 PixiJS AnimatedSprite 或自定义帧播放器。
- 是 Live2D：接入 Live2D Web SDK / pixi-live2d-display 一类方案。
- 是 Spine：接入 PixiJS Spine runtime。

4. 互动动作升级

优先做这些高感知动作：

- 眨眼
- 摇尾巴
- 摸头反应
- 打哈欠
- 开心小跳
- 探头
- 生气炸毛
- 睡觉呼吸
- 收到消息抬头

5. 稳定性回归

每次素材和动作升级后都要回归：

- 待机
- 拖拽
- 摸头
- 长按
- 睡觉 / 唤醒
- 连接测试
- 说话回复
- 设置窗口
- 托盘显示 / 隐藏
